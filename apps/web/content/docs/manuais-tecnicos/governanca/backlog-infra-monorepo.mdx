## Objetivo

Concentrar apenas o backlog ainda nao concluido, priorizando:

- escalabilidade da arquitetura
- reducao de acoplamento
- qualidade de entrega
- preparacao para mobile e backend separado

## Diretriz de execucao

O backlog passa a seguir esta ordem:

1. consolidar o **modular monolith**
2. estabilizar contratos e boundaries
3. extrair o **monorepo estrutural**
4. extrair o **monorepo de dominio**
5. abrir caminho para mobile e API dedicada

Nao vamos tratar workspaces, mobile e API separada como primeiros passos.

## Fase A - Modular Monolith Disciplinado

### EPIC A2 - Actions finas e contratos internos

- [x] `INFRA-006` [Alta] Criar view models e contratos por feature dentro do app atual.
Criterios de aceite:
- dashboard, listagem e detalhes usam o mesmo contrato.
- nenhum componente forte depende de shape improvisado local.
Status de fechamento:
- `tickets` passou a centralizar contratos de listagem, detalhe, paginação, status e dashboard em `src/features/tickets/domain/model.ts`.
- `src/components/platform/tickets/types.ts` virou apenas reexport do dominio da feature, eliminando duplicacao de contratos.
- dashboard e listagem de chamados passaram a consumir o mesmo contrato do modulo.
- painéis fiscais de anexos, crédito presumido e NCM passaram a consumir diretamente os tipos do dominio fiscal, sem tipos locais duplicados.

- [x] `INFRA-007` [Alta] Padronizar gateways/adapters tecnicos em `core/infrastructure`.
Criterios de aceite:
- Zammad, company registry, tax sync e observabilidade seguem contrato explicito.
- pagina nao fala com client tecnico direto.
Status de fechamento:
- `company-registry` ja operava com contrato explicito em `src/core/domain/interfaces/company-registry-gateway.interface.ts`.
- `Zammad` passou a expor contrato em `src/core/domain/interfaces/zammad-gateway.interface.ts`, com adapter tipado em `src/core/infrastructure/gateways/zammad-gateway.ts`.
- observabilidade Zammad passou a expor gateway tipado em `src/core/domain/interfaces/observability-gateway.interface.ts` e `src/core/infrastructure/observability/zammad-observability.ts`.
- `tax sync` passou a usar contrato explicito de payload em `src/features/tax/domain/model.ts`.
- a dashboard page deixou de importar observabilidade tecnica direto de infraestrutura e passou a consumir a feature de tickets.

## Fase B - Monorepo Estrutural

### EPIC B1 - Workspace e shell

- [x] `INFRA-010` [Alta] Estruturar raiz do monorepo com `apps/web` e `packages/*`.
Criterios de aceite:
- `pnpm-workspace.yaml` e `turbo.json` configurados.
- `apps/web` criado sem regressao funcional.
Status de fechamento:
- `turbo.json` criado com tasks base (`build`, `dev`, `lint`, `typecheck`, `test`).
- `apps/web` criado como shell seguro para a futura migração física da aplicação em `INFRA-011`.
- `packages/shared`, `packages/ui` e `packages/contracts` criados como pontos de extração incremental, sem impactar o build atual na raiz.

- [x] `INFRA-011` [Alta] Mover a aplicacao Next.js para `apps/web`.
Criterios de aceite:
- build, docs, auth e app funcionando a partir de `apps/web`.
- imports ajustados sem alias quebrado.
Status de fechamento:
- artefatos do app foram movidos da raiz para `apps/web` (`src`, `content`, `public`, `prisma`, `scripts`, `tests` e configs do Next).
- `package.json` da raiz passou a atuar como workspace shell e delegar scripts para `@dosc-syspro/web`.
- `apps/web/package.json` assumiu os scripts reais da aplicação Next.js.
- `apps/web` ficou como nova raiz funcional do app, preservando o alias `@/*` via `apps/web/tsconfig.json`.

- [x] `INFRA-012` [Alta] Padronizar qualidade compartilhada.
Criterios de aceite:
- ESLint, tsconfig base e convencoes compartilhadas.
- pipeline pronta para workspace.
Status de fechamento:
- `tsconfig.base.json` criado na raiz como base compartilhada do workspace.
- `apps/web/tsconfig.json` passou a estender a base compartilhada.
- `apps/web/.eslintrc.json` criado para consumir a configuracao arquitetural compartilhada da raiz.
- scripts de qualidade `lint`, `typecheck` e `test` foram padronizados em `apps/web/package.json` e delegados pela raiz.
- `turbo.json` ja encontra tasks correspondentes para a fase de workspace.

- [x] `INFRA-012A` [Alta] Alinhar operacao do workspace (package manager, lockfile e deploy).
Criterios de aceite:
- definir oficialmente `npm` ou `pnpm` como gerenciador do workspace
- eliminar inconsistência entre `pnpm-workspace.yaml`, `package-lock.json` e scripts de install/build
- validar install/build/deploy no novo layout com `apps/web`
Status de fechamento:
- `npm` definido como gerenciador oficial do workspace em `package.json`.
- `pnpm-workspace.yaml` removido para eliminar a ambiguidade operacional.
- raiz mantida com `npm workspaces` e `package-lock.json`, alinhando install e scripts ao mesmo ecossistema.
- scripts da raiz seguem delegando para `@dosc-syspro/web` via `npm -w`, mantendo consistência com o fluxo escolhido.
Observacao:
- a validação final de install/build/deploy depende de execução real no ambiente com `npm`, que não está disponível neste runtime local.
