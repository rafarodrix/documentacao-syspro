## Objetivo

Permitir que um unico host fisico, identificado por um unico `RustDesk ID`, publique multiplas leituras de atualizacao Syspro para empresas ou bases diferentes sem duplicar o conceito de maquina no portal.

Exemplo real:

- mesma maquina
- mesmo `RustDesk ID`
- empresa `X` no disco `C:`
- empresa `Y` no disco `D:`

Hoje o modelo remoto e essencialmente `1 host = 1 empresa = 1 leitura operacional`. Esse desenho quebra nesse cenario.

## Decisao

Manter `RemoteHost` como identidade da maquina e introduzir uma entidade filha para leituras multi-bases.

Resumo:

- `RemoteHost` continua sendo a maquina fisica / acesso remoto
- `agentExternalId` continua sendo o `RustDesk ID` da maquina
- `installToken` continua sendo o segredo de bootstrap
- uma nova tabela guarda as leituras por empresa/caminho monitorado

Isso evita:

- duplicar hosts para o mesmo `RustDesk ID`
- conflito de heartbeat entre empresas
- ambiguidade entre maquina fisica e base Syspro

## Modelo proposto

### Entidade atual mantida

`RemoteHost`

Responsabilidade:

- identidade da maquina
- token de bootstrap por host
- `RustDesk ID`
- heartbeat global da maquina
- status operacional do host
- sessoes remotas

### Nova entidade

`RemoteHostSysproUpdate`

Responsabilidade:

- representar cada base/empresa monitorada dentro do mesmo host
- guardar ultimo caminho observado e a ultima data de atualizacao lida pelo agente

Campos sugeridos:

```prisma
model RemoteHostSysproUpdate {
  id String @id @default(cuid())

  hostId String
  host   RemoteHost @relation(fields: [hostId], references: [id], onDelete: Cascade)

  companyId String?
  company   Company? @relation(fields: [companyId], references: [id], onDelete: SetNull)

  companyLabel String
  path String
  lastFileWriteAt DateTime?
  lastHeartbeatAt DateTime

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([hostId])
  @@index([companyId])
  @@unique([hostId, companyLabel, path])
  @@map("remote_host_syspro_update")
}
```

## Regra de negocio

### O que e host

Host continua sendo:

- a maquina fisica
- o `RustDesk ID`
- o alias remoto
- o ponto de acesso rapido

### O que e leitura Syspro

Cada item de `sysproUpdates[]` representa:

- uma empresa logica
- um caminho monitorado
- uma data de atualizacao do executavel ou arquivo observado

### Relacao entre os dois

- `1 RemoteHost` pode ter `N RemoteHostSysproUpdate`
- o heartbeat do host atualiza o host e tambem sincroniza a lista filha

## Contrato da API

### Payload novo (`POST /api/remote/rustdesk/sync`)

Depois do bootstrap, o `sync` autenticado por `agentToken` passa a aceitar:

```json
{
  "agentToken": "agt_xxx",
  "rustdeskId": "164126438",
  "machineName": "SERVIDOR-01",
  "agentVersion": "rustdesk-oss-local",
  "sysproUpdates": [
    {
      "empresa": "Empresa X",
      "caminho": "C:\\syspro\\sysptoserver.exe",
      "ultimaAtualizacao": "2023-10-15 14:30:00"
    },
    {
      "empresa": "Empresa Y",
      "caminho": "D:\\syspro_clienteY\\sysptoserver.exe",
      "ultimaAtualizacao": "2023-10-10 09:15:00"
    }
  ]
}
```

### Compatibilidade

Regras:

- bootstrap inicial permanece em `POST /api/remote/rustdesk/bootstrap` com `installToken`
- ciclo recorrente deve usar apenas `POST /api/remote/rustdesk/sync` com `agentToken`
- se `sysproUpdates` vier, alem do update do host, sincroniza as linhas filhas

## Comportamento da rota de sync

Fluxo recomendado:

1. validar `agentToken`
2. localizar `RemoteHost`
3. atualizar metadados globais do host
4. fazer upsert da lista `sysproUpdates[]`
5. marcar `lastHeartbeatAt` da maquina
6. marcar `lastHeartbeatAt` de cada leitura

### Upsert das leituras

Chave natural sugerida:

- `hostId`
- `companyLabel`
- `path`

Regras:

- se a combinacao existir, atualiza
- se nao existir, cria
- opcionalmente marcar como `stale` no futuro o que deixou de vir no payload

## Impacto no frontend

### Diretorio remoto

A listagem principal deve continuar focada na maquina:

- empresa principal ou nome do agrupamento
- descricao do host
- `RustDesk ID`
- acesso rapido
- visualizar detalhes

Nao deve explodir a listagem principal em varias linhas por base.

### Detalhe do host

A tela `/portal/plataforma-remota/[hostId]` passa a ganhar uma secao:

- `Bases monitoradas`

Colunas sugeridas:

- empresa
- caminho
- ultima atualizacao
- ultimo heartbeat dessa leitura

### Regra visual

Se houver mais de uma leitura:

- exibir badge `Multi-bases`
- mostrar contador `N bases monitoradas`

## Vinculo com empresa

Ha dois modos validos.

### Modo 1

`companyId` opcional por leitura.

Uso:

- quando a empresa da leitura existe no portal e pode ser vinculada

### Modo 2

somente `companyLabel`

Uso:

- quando a leitura vem com nome livre e ainda nao ha vinculo exato

Decisao recomendada:

- manter ambos
- `companyLabel` sempre obrigatorio
- `companyId` opcional numa primeira fase

Assim o agente nao depende de IDs internos do portal para funcionar.

## Ajustes futuros no script

O script deve evoluir para:

- lista configuravel de servidores/bases
- primeiro heartbeat imediato
- tarefa recorrente a cada 5 minutos
- alias visual tipo `SERVIDOR-01 - Multi-Bases`

O agente nao precisa conhecer `companyId`; ele pode enviar `empresa`, `caminho` e `ultimaAtualizacao`.

## Migração incremental

### Fase 1

- criar tabela `RemoteHostSysproUpdate`
- aceitar `sysproUpdates[]` na rota
- manter compatibilidade com payload antigo

### Fase 2

- mostrar badge `Multi-bases` no detalhe
- listar bases monitoradas na tela de host

### Fase 3

- enriquecer matching com `companyId`
- criar regra de stale/remocao logica para leituras que deixaram de ser reportadas

## O que nao fazer

- nao criar um `RemoteHost` por empresa quando a maquina fisica for a mesma
- nao usar `RustDesk ID + empresa` como identidade primaria do host
- nao sobrescrever a leitura de uma empresa com a de outra no mesmo heartbeat

## Resumo executivo

O desenho correto e:

- `RemoteHost` representa a maquina
- `RemoteHostSysproUpdate` representa cada base/empresa monitorada dentro dela

Isso preserva o fluxo atual do remoto, evita duplicidade de hosts e prepara o portal para receber heartbeats multi-bases sem perder a semantica operacional do `RustDesk ID`.
