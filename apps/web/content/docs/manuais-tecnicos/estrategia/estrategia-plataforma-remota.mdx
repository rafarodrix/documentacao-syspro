Este documento consolida a direcao tecnica e o estado atual da plataforma remota da Trilink, com foco em acesso remoto seguro, integracao com atendimento, segregacao por empresa e evolucao gradual para backup, cofres de credenciais e auditoria operacional.

## Resumo executivo

- Engine remota recomendada para MVP: **RustDesk self-hosted**.
- Integracao operacional inicial: **Zammad + RustDesk**.
- Cofre de credenciais planejado: **HashiCorp Vault**.
- Banco legado Firebird:
  - **`gbak`** como padrao de backup logico.
  - **`nbackup`** em cenarios especificos de backup fisico/incremental.

<Callout type="info" title="Diretriz pratica">
  A base atual prioriza operacao e governanca antes de sofisticacao. O objetivo imediato nao e browser access completo, e sim controle de hosts, sessoes, vinculo com ticket e tenant enforcement por empresa.
</Callout>

## Estado atual da implementacao

Ja existe no codigo:

- modelos persistidos de `RemoteHost` e `RemoteSession`
- `companyId` explicito em host e sessao
- `ticketId` e `ticketNumber` explicitos em `RemoteSession`
- rotas iniciais:
  - `GET/POST /api/remote/hosts`
  - `GET/POST /api/remote/sessions`
  - `POST /api/remote/sessions/:id/start`
  - `POST /api/remote/sessions/:id/stop`
- integracao inicial com `Zammad`:
  - deep-link `rustdesk://`
  - webhook que cria/encerra `RemoteSession`
  - nota interna automatica no Zammad ao iniciar/encerrar sessao
- enforcement de tenant:
  - `ADMIN`, `SUPORTE`, `DEVELOPER` com visao global
  - `CLIENTE_ADMIN` com escopo apenas da propria empresa

Tambem ja existe separacao de experiencia:

- `Configuracoes > Acesso Remoto`
  - cadastro de hosts
  - solicitacao manual de sessao
  - configuracao operacional
- `/app/plataforma-remota`
  - diretorio operacional dos hosts/clientes ja configurados
  - acesso ao detalhe do host
  - ponto de abertura do acesso remoto

Tambem ja existe uma matriz compartilhada de status operacional no dominio:

- `ONLINE`
- `RECENT`
- `OFFLINE`
- `MISCONFIGURED`
- `SESSION_BUSY`

## Arquitetura aplicada

### Camadas

1. **Portal Web (Next.js)**
2. **API interna via App Router**
3. **Orquestracao remota baseada em host + sessao**
4. **Integracao com Zammad**
5. **Expansao futura para Vault, backup e auditoria ampliada**

### Modulos principais

- gestao de hosts remotos por empresa
- sessao remota com status e historico
- integracao atendimento -> acesso remoto
- trilha minima por ticket/sessao
- segregacao por tenant

## Politica de acesso

- `ADMIN`, `SUPORTE` e `DEVELOPER` possuem visao global da plataforma remota.
- `CLIENTE_ADMIN` deve acessar apenas hosts e sessoes da propria empresa.
- `CLIENTE_USER` nao participa do fluxo remoto no MVP.

<Callout type="warning" title="Regra obrigatoria">
  Toda query, rota e acao operacional nova da plataforma remota deve preservar a mesma regra de tenant aplicada hoje no shell, na listagem, no detalhe do host e nas rotas de `hosts` e `sessions`.
</Callout>

## Fluxos criticos

### 1) Cadastro e organizacao de host

1. operador acessa `Configuracoes > Acesso Remoto`
2. escolhe a empresa
3. cadastra host com nome, ambiente e provider
4. host passa a aparecer no diretorio operacional

### 2) Operacao remota por host

1. usuario acessa `/app/plataforma-remota`
2. visualiza apenas hosts no proprio escopo
3. abre o detalhe do host
4. dispara o acesso remoto se existir `rustdesk_id`

### 3) Sessao remota vinculada a ticket

1. sessao pode ser aberta manualmente ou por webhook do `Zammad`
2. `RemoteSession` registra `ticketId` e `ticketNumber`
3. regra operacional atual:
   - um mesmo ticket pode ter varias sessoes ao longo do tempo
   - nao pode existir sessao aberta duplicada para o mesmo ticket/host
4. ao iniciar e encerrar:
   - o sistema registra nota interna no Zammad

## Integracao Zammad + RustDesk

### Objetivo

Conectar atendimento e acesso remoto sem depender de plugin proprietario:

- `Zammad` como origem do ticket
- `RustDesk` como engine remota
- plataforma remota como camada de policy, trilha e segregacao por empresa

### O que ja esta implementado

- rota de deep-link:
  - `GET /api/integrations/zammad/rustdesk-link/:ticketId`
- webhook de integracao:
  - `POST /api/integrations/zammad/webhook`
- criacao/encerramento de `RemoteSession` a partir de evento do ticket
- comentario interno automatico no Zammad ao iniciar/encerrar sessao

### O que ainda falta

- validar payload real do `Zammad`
- configurar custom link real no `Zammad`
- decidir se o host sera sempre resolvido por `rustdesk_id` ou se havera escolha explicita no atendimento
- persistir auditoria por sessao
- endurecer politicas de deduplicacao e fechamento operacional

### Padrao recomendado

1. armazenar `rustdesk_id` no contexto do cliente/ticket
2. abrir acesso via `rustdesk://<id>`
3. vincular o ticket a `RemoteSession`
4. registrar nota interna automatica no atendimento

## Validacao OSS-first recomendada

### 1) Validar cadastro e host operacional

1. criar host em `Configuracoes > Acesso Remoto`
2. confirmar presenca de:
   - `RustDesk ID`
   - `installToken`
   - descricao operacional
3. abrir `/app/plataforma-remota`
4. confirmar listagem, busca e detalhe do host
5. validar abertura via `rustdesk://<id>`

### 2) Validar auto-registro do agente

Exemplo de payload para `POST /api/remote/agents/register`:

```json
{
  "installToken": "seed-token-<companyId>-prod",
  "rustdeskId": "123456789",
  "machineName": "ERP-MATRIZ-01",
  "agentVersion": "1.4.0",
  "environment": "Producao"
}
```

Resultado esperado:

- `agentExternalId` preenchido
- `machineName` preenchido
- `agentVersion` preenchido
- `lastHeartbeatAt` atualizado
- `status` ajustado para `ACTIVE`

### 3) Validar heartbeat

Exemplo de payload para `POST /api/remote/agents/heartbeat`:

```json
{
  "installToken": "seed-token-<companyId>-prod",
  "rustdeskId": "123456789",
  "machineName": "ERP-MATRIZ-01",
  "agentVersion": "1.4.0"
}
```

Resultado esperado:

- `lastHeartbeatAt` atualizado a cada chamada
- metadata do agente refletida no detalhe do host
- diretorio operacional mostrando heartbeat recente

### 4) Validar sessao remota

1. criar sessao manual
2. iniciar sessao
3. encerrar sessao
4. confirmar transicao:
   - `REQUESTED -> STARTED -> ENDED`
5. confirmar bloqueios:
   - host nao deve ser excluido com sessao aberta
   - sessao duplicada para mesmo ticket/host deve ser recusada

## Experiencia atual do produto

### Configuracao

- `Configuracoes > Acesso Remoto`
- usada para:
  - cadastro de host
  - solicitacao manual de sessao
  - preparacao operacional

### Operacao

- `/app/plataforma-remota`
- usada para:
  - listar hosts/clientes configurados
  - exibir `id`, empresa e descricao operacional
  - acessar o detalhe do host
  - abrir o acesso remoto

### Fluxo OSS-first do agente

Ja existe suporte para:

- `POST /api/remote/agents/register`
- `POST /api/remote/agents/heartbeat`
- script PowerShell do agente
- download de `.ps1` por host
- regeneracao automatica de `installToken` legado

O que ainda depende de ambiente real:

- validar heartbeat periodico em maquina real
- endurecer o segredo de `installToken`
- substituir o bootstrap cru por `agentToken` em operacao continuada
