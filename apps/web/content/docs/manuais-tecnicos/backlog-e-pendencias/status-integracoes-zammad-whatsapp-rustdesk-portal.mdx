## Escopo

Este documento consolida o estado real da integracao entre:

- Zammad
- WhatsApp (Evolution)
- RustDesk
- Portal

## Resumo executivo (2026-04-04)

- Zammad: **producao** com cache em banco, sync worker e webhook oficial com HMAC.
- WhatsApp: **producao parcial com backend consolidado** (`apps/api` para inbound/outbound/listagem/resolucao/vinculo/start outbound; `apps/web` como UI/admin).
- RustDesk: **MVP funcional** com `discover -> bootstrap -> sync -> ack` em codigo; validacao E2E em campo ainda pendente.
- Portal: **operacao principal** de tickets, remoto e configuracoes.

## Topologia de borda (infra atual)

- `Nginx Proxy Manager` instalado no mesmo servidor de `RustDesk` e `Evolution Go`.
- painel operacional atual do proxy:
  - `http://85.239.248.141:81/nginx/proxy`
- mapeamento de dominio em uso:
  - `acesso.<dominio>` -> RustDesk (VPS)
  - `api.<dominio>` -> Evolution Go (VPS)
- mapeamento previsto para backend dedicado:
  - `backend.<dominio>` -> `apps/api` (Railway)

  Nao expor a API da Evolution por IP:porta em ambiente publico. Manter publicacao via proxy reverso com TLS.

## Estado por frente

### Zammad

- Em producao:
  - `POST /api/platform/zammad/webhook` (HMAC)
  - alias `POST /api/integrations/zammad/webhook`
  - `POST /api/platform/zammad/sync`
  - `ZammadTicketCache` + `ZammadSyncState`
- Mudanca aplicada:
  - endpoint legado `POST /api/webhooks/zammad` desativado (HTTP 410).

### WhatsApp (Evolution)

- Ja migrado para `apps/api`:
  - `POST /api/webhooks/evolution` (orquestracao inbound)
  - `GET /api/conversations?filter=ATENDENDO|ESPERA`
  - `GET /api/conversations/:conversationId`
  - `GET /api/conversations/:conversationId/messages`
  - `POST /api/conversations/send`
  - `POST /api/conversations/resolve`
  - `POST /api/conversations/link`
  - `POST /api/conversations/start-outbound`
- `apps/web` mantido como camada UI/admin:
  - rota interna `POST /api/webhooks/evolution` apenas como proxy para o backend (transicao)
  - tela `/portal/conversas` consumindo casos de uso via `apps/api`
- Seguranca de servico entre `web -> api`:
  - header obrigatorio `x-internal-api-key`
  - chave em `INTERNAL_API_KEY` nos dois runtimes (`apps/web` e `apps/api`)
- Contrato compartilhado implementado:
  - `@dosc-syspro/contracts` com schema de envelope de webhook (`event` + `data`)
  - padronizacao de env:
    - `EVOLUTION_API_URL`
    - `EVOLUTION_API_KEY`
    - `EVOLUTION_INSTANCE`
    - `EVOLUTION_WEBHOOK_SECRET`
- Pendente:
  - extrair bridge para `apps/whatsapp-bridge`
  - fila operacional dedicada (Redis/BullMQ)
