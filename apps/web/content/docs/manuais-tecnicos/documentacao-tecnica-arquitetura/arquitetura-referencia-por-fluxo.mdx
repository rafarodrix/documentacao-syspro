## Objetivo

Consolidar fluxos de referencia para as principais areas do produto, com visao ponta a ponta:

- `UI -> API -> Core -> Database`

Esta pagina serve como base para novos modulos e para revisoes de arquitetura.

## Fluxo de Cadastro

### Escopo

- Gestao de entidades basicas (empresa, usuarios, tabelas e cadastros operacionais).

### Diagrama textual

```text
[UI - apps/web]
  Paginas de cadastro, formularios, validacoes de interface
      ->
[API - packages/api]
  Routers tRPC, validacao de entrada, contexto de tenant, autorizacao
      ->
[Core - packages/core]
  Casos de uso de cadastro, regras de dominio, validacoes de negocio
      ->
[Database - packages/database]
  Prisma Client, persistencia e transacoes
```

### Regras chave

1. Validacao de UX fica na UI; validacao de negocio fica no Core.
2. API nao pode conter regra de dominio complexa.
3. Escrita no banco apenas via camada autorizada.

## Fluxo de Suporte

### Escopo

- Processos operacionais, atendimento, integracoes e playbooks internos.

### Diagrama textual

```text
[UI - apps/web]
  Painel de suporte, consultas operacionais, acionamento de processos
      ->
[API - packages/api]
  Endpoints/routers para suporte, controle de acesso por perfil
      ->
[Core - packages/core]
  Regras de triagem, criterios de prioridade, automacoes de processo
      ->
[Database - packages/database]
  Registro de chamados, eventos e trilha de auditoria
```

### Regras chave

1. Todo fluxo critico deve gerar rastreabilidade (log/auditoria).
2. Integracoes externas devem ser isoladas em adapters.
3. Fluxos de suporte precisam fallback seguro em caso de erro externo.

## Fluxo de Financeiro

### Escopo

- Contas a pagar/receber, conciliacao e consultas de indicadores.

### Diagrama textual

```text
[UI - apps/web]
  Telas de lancamento, consulta e conciliacao financeira
      ->
[API - packages/api]
  Orquestracao de comandos e consultas, enforce de tenant e role
      ->
[Core - packages/core]
  Regras de fechamento, calculos, consistencia contabil/financeira
      ->
[Database - packages/database]
  Persistencia de lancamentos, saldos e historico de alteracoes
```

### Regras chave

1. Operacoes sensiveis exigem controle de permissao por role.
2. Calculos financeiros devem ser centralizados no Core.
3. Atualizacoes concorrentes precisam estrategia de consistencia.

## Checklist para novos fluxos

- Definir fronteira clara entre UI, API, Core e Database.
- Especificar contratos de entrada/saida em `packages/contracts`.
- Garantir observabilidade minima (logs e eventos essenciais).
- Validar regras de autorizacao por perfil e tenant.
- Documentar cenarios de erro e fallback.

## Anti-padroes por fluxo

### Cadastro

- Validar regra de negocio apenas na UI.
- Executar escrita sem passar por caso de uso no Core.
- Permitir mutacoes sem verificacao de tenant/perfil.

### Suporte

- Integrar servico externo diretamente na tela.
- Nao registrar eventos criticos para auditoria.
- Tratar erro externo sem fallback operacional.

### Financeiro

- Calcular valores sensiveis fora do Core.
- Nao usar estrategia de consistencia para concorrencia.
- Expor operacao critica sem controle de role.
