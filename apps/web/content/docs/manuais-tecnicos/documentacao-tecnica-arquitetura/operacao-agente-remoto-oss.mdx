Este manual descreve o fluxo operacional vigente do agente remoto no modelo **OSS-first**, em que o portal da Trilink continua sendo a fonte de verdade para host, heartbeat, instalacoes monitoradas e metadata do agente, enquanto o **RustDesk self-hosted** fica responsavel pela conectividade remota.

## Objetivo

Padronizar como uma maquina remota deve ser:

- descoberta sem pre-cadastro
- vinculada a empresa/host quando necessario
- registrada diretamente em host ja existente
- mantida online via heartbeat recorrente
- validada antes do uso pelo suporte

## Premissas

- a operacao diaria acontece em `/portal/plataforma-remota`
- as configuracoes globais do modulo remoto ficam em `/portal/configuracoes?tab=remote`
- o portal expoe:
  - `GET /api/remote/agents/discovery-script`
  - `POST /api/remote/agents/discover`
  - `POST /api/remote/rustdesk/bootstrap`
  - `POST /api/remote/rustdesk/sync`
  - `POST /api/remote/rustdesk/ack`
  - `GET /api/remote/rustdesk/client-profile`
  - `POST /api/remote/discovered-hosts/:id/link`
- a rota de descoberta aplica protecao anti-abuso na camada HTTP:
  - rate limit por IP em `POST /api/remote/agents/discover` (5 req/min)
  - resposta `429` quando exceder a janela
- o `RustDesk ID` deve ser persistido sem espacos
- a leitura operacional do host hoje segue a matriz:
  - `ONLINE`
  - `RECENT`
  - `OFFLINE`
  - `MISCONFIGURED`
  - `SESSION_BUSY`

<Callout type="warning" title="Fonte de verdade">
  No recorte OSS-first, o portal da Trilink continua sendo a fonte de verdade para `installToken`, metadata do agente, heartbeat, multi-instalacoes e relacao entre empresa, host e sessao. O RustDesk nao substitui essa camada.
</Callout>

## Fluxo oficial consolidado

Sequencia oficial unica para operacao de agente remoto:

1. `POST /api/remote/agents/discover` (triagem inicial sem pre-cadastro)
2. `POST /api/remote/rustdesk/bootstrap` (bootstrap autenticado por host e emissao de `agentToken`)
3. `POST /api/remote/rustdesk/sync` (heartbeat operacional recorrente com compliance e fila de comandos)
4. `POST /api/remote/rustdesk/ack` (retorno de execucao dos comandos recebidos em `commandQueue`)

<Callout type="info" title="Regra operacional">
  O fluxo legado `agents/register` e `agents/heartbeat` nao faz parte do processo oficial. Em campo, use apenas `discover` para triagem e `bootstrap/sync/ack` para operacao autenticada.
</Callout>

## Fluxos oficiais

Hoje existem **dois fluxos oficiais** de instalacao.

### 1. Descoberta sem pre-cadastro

Usado quando a maquina ainda nao existe no portal.

<Steps>
  <Step>
    ### 1. Baixar script padrao
    O operador baixa o script em `GET /api/remote/agents/discovery-script` pela tela `/portal/plataforma-remota`.
  </Step>
  <Step>
    ### 2. Executar na maquina
    O script instala/atualiza o RustDesk, aplica a configuracao do servidor proprio, tenta autoelevacao, envia a primeira descoberta e agenda heartbeat recorrente.
  </Step>
  <Step>
    ### 3. Entrada em triagem
    A maquina aparece em `Maquinas pendentes de vinculacao` no diretorio remoto, ainda como descoberta sem host operacional definitivo.
  </Step>
  <Step>
    ### 4. Vinculacao
    O operador escolhe a empresa, define o nome do host e converte a descoberta em host operacional via `POST /api/remote/discovered-hosts/:id/link`.
  </Step>
  <Step>
    ### 5. Operacao do suporte
    O suporte passa a operar o host ja vinculado pelo diretorio e pelo detalhe da maquina.
  </Step>
</Steps>

### 2. Instalacao por host ja cadastrado (fluxo oficial)

Usado quando o host ja existe no portal e o agente deve entrar direto no cadastro correto.

<Steps>
  <Step>
    ### 1. Cadastro do host
    O operador cria ou revisa o host em `/portal/plataforma-remota` ou em `/portal/configuracoes?tab=remote`.
  </Step>
  <Step>
    ### 2. Download do instalador do host
    O operador baixa o `.ps1` do host em `GET /api/remote/hosts/:id/installer`.
  </Step>
  <Step>
    ### 3. Auto-registro
    O agente envia `installToken`, `rustdeskId`, `machineName` e versao atual para `POST /api/remote/rustdesk/bootstrap` e recebe `agentToken`.
  </Step>
  <Step>
    ### 4. Sync recorrente
    O agente envia chamadas periodicas para `POST /api/remote/rustdesk/sync` com `agentToken`, mantendo heartbeat e compliance. O envio de `sysproUpdates[]` deve ser incremental (apenas quando houver mudanca de versao/status/base); sem mudanca, enviar array vazio.
  </Step>
  <Step>
    ### 5. Ack de comandos
    Quando o portal entregar comandos na `commandQueue`, o agente executa e devolve resultado para `POST /api/remote/rustdesk/ack`.
  </Step>
  <Step>
    ### 6. Operacao do suporte
    O suporte consulta `/portal/plataforma-remota`, usa `Acesso rapido` ou abre o detalhe do host antes da sessao.
  </Step>
</Steps>

## Payload minimo recomendado

### Descoberta

Endpoint:

```text
POST /api/remote/agents/discover
```

Payload recomendado:

```json
{
  "discoveryToken": "REMOTE_DISCOVERY_TOKEN",
  "rustdeskId": "21187620068",
  "machineName": "ERP-MATRIZ-01",
  "agentVersion": "rustdesk-oss-discovery",
  "serviceStatus": "running",
  "sysproUpdates": [
    {
      "empresa": "EMPRESA X",
      "caminho": "C:\\syspro\\sysptoserver.exe",
      "ultimaAtualizacao": "2026-03-25 10:00:00"
    }
  ]
}
```

Resultado esperado:

- descoberta criada ou atualizada
- `machineName` preenchido
- `agentExternalId` preenchido quando o `RustDesk ID` vier valido
- `serviceStatus` e `sysproUpdates[]` persistidos no host descoberto ou no host ja vinculado

Contrato de transicao (`discover.v2`):

- `bootstrapFlow = pending_link`: triagem ativa, pode manter heartbeat de descoberta ate o vinculo manual.
- `bootstrapFlow = linked_host_detected`: host ja vinculado e operando em `bootstrap/sync`; discovery heartbeat deve ser interrompido.
- `bootstrapFlow = host_installer_required`: host vinculado sem credencial valida; operador deve baixar o `.ps1` dedicado do host e executar `bootstrap`.
- `transition.allowDiscoveryHeartbeat` e o campo formal que define se o script deve agendar ou nao a rotina recorrente de `discover`.

### Bootstrap inicial em host operacional

Endpoint:

```text
POST /api/remote/rustdesk/bootstrap
```

Exemplo:

```json
{
  "installToken": "rhost_abc123",
  "rustdeskId": "21187620068",
  "machineName": "ERP-MATRIZ-01",
  "agentVersion": "rustdesk-oss-local",
  "environment": "Producao"
}
```

Resultado esperado:

- `agentExternalId` preenchido
- `machineName` preenchido
- `agentVersion` preenchido

### Sync operacional

Endpoint:

```text
POST /api/remote/rustdesk/sync
```

Exemplo:

```json
{
  "agentToken": "agt_xxx",
  "rustdeskId": "21187620068",
  "machineName": "ERP-MATRIZ-01",
  "agentVersion": "rustdesk-oss-local",
  "currentAlias": "EMPRESA X | ERP-MATRIZ-01",
  "currentVersion": "1.4.6",
  "serverHost": "acesso.trilinksoftware.com.br",
  "apiHost": "acesso.trilinksoftware.com.br",
  "publicKey": "6FpnQH+KbbpX0qw6XxF0xqnIO0QnHImwbvQ5Lv7q6gU=",
  "serviceStatus": "running",
  "sysproUpdates": [
    {
      "empresa": "EMPRESA X",
      "caminho": "C:\\syspro\\sysptoserver.exe",
      "ultimaAtualizacao": "2026-03-25 10:00:00"
    }
  ]
}
```

Resultado esperado:

- `lastHeartbeatAt` atualizado
- `serviceStatus` refletido no detalhe do host
- instalacoes monitoradas atualizadas sem duplicar hosts
- `compliance` retornado pelo portal
- `commandQueue` entregue para execucao local

### Otimizacao de payload no sync (obrigatoria)

Para reduzir overhead de rede em hosts com muitas bases monitoradas:

- o agente deve manter hash local do snapshot de `sysproUpdates[]` (por exemplo, hash de empresa + caminho + ultimaAtualizacao)
- no ciclo de `sync`, enviar `sysproUpdates[]` completo somente quando o hash mudar
- se nao houve mudanca desde o ultimo envio, enviar `sysproUpdates: []`
- o heartbeat e os campos operacionais (`serviceStatus`, `agentVersion`, `machineName`, compliance) continuam sendo enviados normalmente
- politica diaria: enviar ao menos 1 snapshot completo por dia por host (primeiro `sync` apos ligar e primeiro `sync` apos meia-noite)
- apos cumprir o snapshot diario, pode enviar `sysproUpdates: []` no restante do dia enquanto nao houver mudanca

Exemplo de `sync` sem mudancas de instalacao:

```json
{
  "agentToken": "agt_xxx",
  "rustdeskId": "21187620068",
  "machineName": "ERP-MATRIZ-01",
  "agentVersion": "rustdesk-oss-local",
  "serviceStatus": "running",
  "sysproUpdates": []
}
```

## Scripts oficiais

### Script padrao

Origem oficial:

```text
GET /api/remote/agents/discovery-script
```

Observacoes:

- e o **script oficial de producao**
- usa `REMOTE_DISCOVERY_TOKEN` do ambiente
- consome as configuracoes globais do modulo remoto:
  - servidor RustDesk
  - config exportada
  - chave publica
  - versao alvo
  - intervalo de heartbeat
  - senha padrao

### Script estatico em `public`

Arquivo:

```text
public/downloads/trilink-instalador-padrao.ps1
```

Uso recomendado:

- documentacao interna
- referencia tecnica
- nao tratar como fonte oficial quando houver divergencia com a rota dinamica

<Callout type="info" title="Regra pratica">
  Em producao, sempre priorize o script baixado pela rota dinamica. O arquivo estatico em `public` pode ficar defasado se as configuracoes globais do modulo remoto mudarem.
</Callout>

## Checklist de validacao

### Descoberta

- script padrao baixado pela rota dinamica
- primeira chamada para `discover` concluida
- maquina aparece em `Maquinas pendentes de vinculacao`
- vinculacao manual concluida sem erro
- validar que a maquina nao entra em loop de `discover` (evitar `429` recorrente)

### Host operacional

- host criado com empresa correta
- `installToken` presente quando o fluxo for por host
- `RustDesk ID` salvo sem espacos
- `machineName` e `agentVersion` preenchidos

### Heartbeat

- heartbeat periodico ativo
- `serviceStatus` atualizado
- `sysproUpdates[]` enviado apenas quando houver mudanca (array vazio quando nao houver alteracao)

### Operacao

- host aparece em `/portal/plataforma-remota`
- busca por empresa principal e empresas secundarias funciona
- `Acesso rapido` funciona
- detalhe do host mostra nome da maquina, heartbeat e instalacoes

### Sessao

- sessao manual criada
- transicao `REQUESTED -> STARTED -> ENDED`
- bloqueio de exclusao de host com sessao aberta
- bloqueio de sessao duplicada por ticket/host

## Problemas comuns

### `403` na descoberta

O `discoveryToken` enviado nao bate com `REMOTE_DISCOVERY_TOKEN`.

Tratamento:

- baixar novamente o script pela rota dinamica
- validar a variavel de ambiente
- evitar usar copia antiga do script

### `429` na descoberta

A rota `POST /api/remote/agents/discover` aplica rate limit por IP para reduzir risco de abuso/lixo de banco.

Regra atual:

- maximo de 5 requisicoes por minuto por IP

Tratamento:

- validar se o agente entrou em loop de chamadas
- reduzir frequencia de retries no script
- aguardar janela de rate limit e reenviar

### `405` na chamada do agente

Historicamente isso aconteceu quando `middleware` interceptava `/api/remote/agents/*`.

Tratamento:

- validar se a rota do agente continua publica no middleware
- confirmar que a chamada chega direto em `discover`, `bootstrap`, `sync` ou `ack`

### `Impossivel conectar-se ao servidor remoto`

Normalmente indica URL errada para a API do portal.

Tratamento:

- usar `https://ajuda.trilinksoftware.com.br`
- nao apontar a API do agente para o host do servidor RustDesk se ele nao hospeda o portal Next

### Sem heartbeat

O agente nao registrou atividade recente no portal.

Tratamento:

- validar script/tarefa agendada
- validar conectividade com o portal
- validar `installToken` ou `discoveryToken`
- validar permissao do PowerShell

### `about:blank#blocked`

O navegador bloqueou o protocolo externo `rustdesk://`.

Tratamento:

- testar `rustdesk://<id>` no `Win + R`
- copiar o `RustDesk ID` e abrir manualmente no app

## Arquitetura consolidada do modulo remoto

### Diretriz de plataforma

- o portal governa host, sessao, autorizacao e auditoria
- RustDesk fornece conectividade, nao dominio de negocio
- segregacao por tenant e obrigatoria em consulta, acao e sessao
- a operacao atual de API remota esta em `apps/web/src/app/api/remote/**/route.ts`
- a camada `packages/api` ainda nao centraliza o modulo remoto

### Multi-bases no mesmo host

Modelo consolidado:

- `RemoteHost` representa a maquina fisica e o acesso remoto
- `RemoteHostSysproUpdate` representa cada empresa/base monitorada na mesma maquina
- chave natural de sincronizacao: `hostId + companyLabel + path`
- nao duplicar host por empresa quando o `RustDesk ID` for da mesma maquina

## Itens enviados para operacao (estado atual)

- fluxo oficial `discover -> bootstrap -> sync -> ack` ativo
- protecao anti-abuso no `discover` com rate limit de 5 req/min por IP
- `agentToken` emitido no bootstrap e usado no ciclo recorrente
- fila de comandos operacionais com `ack` do agente
- compliance de cliente RustDesk persistido no host
- sincronizacao de `sysproUpdates[]` por host e por leitura filha
- envio incremental de `sysproUpdates[]` no `sync` para reduzir payload em ciclos recorrentes
- configuracao global do modulo remoto aplicada nos scripts dinamicos

## Backlog tecnico unico do remoto

### Prioridade alta

1. Extrair dominio e servicos remotos para package compartilhado
- status atual: **parcialmente concluido**
- `packages/remote-domain` ja concentra casos de uso e contratos de `discover/bootstrap/sync/ack`
- pendente mover o residuo de orquestracao/persistencia dos adapters HTTP para gateways/repositorios de dominio
- pendente conectar o mesmo dominio em `apps/api` e preparar contrato final para `apps/mobile`
- manter independencia de framework (sem dependencia de Next.js no dominio)

2. Endurecimento de seguranca operacional
- reforcar politica de token e rotacao/rebootstrap em campo
- blindar exposicao de segredo em logs e scripts
- endurecer validacoes de entrada e contratos de payload

3. Isolamento e autorizacao por tenant
- consolidar guardas por `companyId` nas operacoes sensiveis
- padronizar checagem de escopo em host/sessao/comando

### Prioridade media

4. Introduzir `remoteRouter` em `packages/api`
- consumir os casos de uso extraidos do package de dominio
- manter compatibilidade com endpoints HTTP atuais durante transicao
- reduzir divergencia entre camada HTTP e router compartilhado

5. Contrato remoto para cliente mobile
- versionar DTOs de request/response
- consolidar contrato estavel de `bootstrap/sync/ack/discover`
- separar claramente fluxo de host script e fluxo de app mobile

6. Observabilidade e performance
- ampliar trilha de auditoria de sessao/comando
- evoluir politica de cache para payloads estaveis (`client-profile`)
- revisar ritmo de heartbeat e protecoes de carga conforme volume real
