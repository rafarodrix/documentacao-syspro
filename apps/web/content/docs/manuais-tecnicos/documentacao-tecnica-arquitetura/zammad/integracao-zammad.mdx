## Objetivo

Registrar como o projeto usa o Zammad hoje, documentar a arquitetura de resilência e evitar que decisões antigas sobre SSO, frontend acoplado ou consultas redundantes voltem a orientar implementações novas.

## Papel atual do Zammad

Hoje o Zammad é tratado como:

- **Sistema de atendimento e ticketing** — origem de verdade para tickets e interações com clientes
- **Provider de eventos** — webhooks para operações remotas (deep-link, criar sessões)
- **Adapter consultado** — via gateway dedicado na camada de infraestrutura, nunca diretamente
- **Origem de observabilidade** — saúde das rotas rastreada por métricas estruturadas

Hoje o Zammad **não** é tratado como:

- Fonte principal de autenticação do portal
- Backend central da aplicação
- Lugar para concentrar regra de negócio da Trilink
- Página ou componente consultando diretamente (sem adapter)

<Callout type="info" title="Diretriz vigente">
  O portal segue com autenticação própria e usa o Zammad como integração operacional. Qualquer acoplamento adicional deve passar por contrato e adapter da feature de tickets. Resiliência é obrigatória: contingência, cache e circuit breaker são da arquitetura, não da aplicação.
</Callout>

### O que já existe no código

- **Gateway dedicado** — `ZammadGateway` para consultas e mutações, nunca chamadas diretas
- **HTTP client robusto** — retry automático, circuit breaker, timeout e cache com stale fallback
- **Cache operacional local** — em memória com TTL per-query, permite degradação até 15 minutos atrás
- **Sincronização proativa** — endpoint `/api/platform/zammad/sync` para preencher cache periodicamente
- **Webhooks bidirecionais** — `/api/integrations/zammad/webhook` recebe eventos para criar sessões remotas
- **Integração com módulo remoto** — `zammad-integration.ts` resolve deep-links e cria sessões autenticadas
- **Observabilidade estruturada** — métricas por rota, rastreamento de estale data e health checks
- **Cache de usuários** — em memória com TTL 5 minutos para `getUserIdByEmail`

## Estratégia de resiliência

### Retry com exponential backoff

Toda requisição HTTP para o Zammad passa por retry automático:

- **Max attempts:** 3 (configurável via `ZAMMAD_RETRY_MAX_ATTEMPTS`)
- **Base delay:** 400ms (configurável via `ZAMMAD_RETRY_BASE_DELAY_MS`)
- **Estratégia:** exponential backoff — delay = base × 2^(tentativa-1)
- **Retryable:** HTTP 429 (rate limit) e 5xx (erro servidor)
- **Não retryável:** 4xx (erro cliente), timeout na última tentativa

```javascript
// Exemplo de delays: 400ms, 800ms, 1600ms
```

### Circuit breaker por rota

Quando uma rota falha repetidamente, um circuit breaker a isola temporariamente:

- **Ativação:** Erro na última tentativa de retry
- **Cooldown:** 20 segundos (configurável via `ZAMMAD_CIRCUIT_COOLDOWN_MS`)
- **Comportamento:** Requisições para rota aberta caem imediatamente no fallback
- **Rastreamento:** Por `routeKey` (exemplo: `"search-operational"`, `"sync-worker"`)
- **Configuração:** Ativar via `ZAMMAD_ENABLE_IN_MEMORY_CIRCUIT_BREAKER=true`

### Timeout

- **Padrão:** 10 segundos (configurável via `ZAMMAD_TIMEOUT_MS`)
- **Mecanismo:** AbortController do Fetch API
- **Comportamento:** Timeout dispara fallback se estiver disponível

### Cache com stale fallback

O cliente mantém um cache em memória com estratégia sofisticada:

1. **Cache fresco** — Requisições GET são armazenadas localmente
2. **Degradação ativa** — Se o Zammad falha e há cache local com menos de 15 minutos (configurável via `ZAMMAD_FALLBACK_MAX_STALE_MINUTES`), retorna cache com marcação de `stale`
3. **TTL por operação** — Cada query pode estar seu próprio `cacheTtlSeconds`
4. **Rastreamento** — Router mantém registro de qual cache está em estado `stale` para observabilidade

Exemplo:
```javascript
// Buscar tickets com cache de 30 segundos
const tickets = await ZammadGateway.searchOperationalTicketsPage(
  query, 
  { cacheTtlSeconds: 30, routeKey: "search-operational" }
);
// Se falha antes de 30s, cache local ainda vale
// Se falha depois de 30s, cache de até 15min atrás é usado com marcação 'stale'
```

### Sincronização proativa

Para garantir cache atualizado e reduzir pressão em picos:

- **Endpoint:** `POST /api/platform/zammad/sync`
- **Frequência:** Executar periodicamente (recomendado: a cada 5 minutos)
- **Paginação:** Configurável via `ZAMMAD_SYNC_PAGES` e `ZAMMAD_SYNC_PER_PAGE`
- **Autorização:** Token do header (não usa sessão web)
- **Comportamento:** Preenche cache de todos os tickets operacionais

## Decisões práticas para novas implementações

### Leitura operacional

1. **Uma consulta principal por contexto** — Evitar N+1 e consultas desnecessárias
   ```typescript
   // ✅ BOM: Uma consulta que traz tudo
   const tickets = await gateway.searchOperationalTicketsPage(query);
   
   // ❌ RUIM: Chamar duas vezes
   const tickets = await gateway.searchOperationalTicketsPage(query);
   const count = await gateway.getTicketCount(query); // redundante
   ```

2. **Usar cache local quando possível** — Não mascarar erro como lista vazia
   ```typescript
   // ✅ BOM: Falha e retorna cache de contingência
   const tickets = await ZammadGateway.getTicketsForCustomerEmailsPaged(emails, {
     cacheTtlSeconds: 60,
     routeKey: "list-customer-tickets"
   });
   // Se Zammad falha, retorna cache até 15 minutos atrás com status 'stale'
   
   // ❌ RUIM: Retornar lista vazia quando há fallback
   try {
     return await fetchZammed(...);
   } catch {
     return []; // Perdeu dados que poderiam estar em cache
   }
   ```

3. **Designar um `routeKey` semanticamente** — Para observabilidade
   ```typescript
   // ✅ BOM: Nome descritivo
   await ZammadGateway.searchOperationalTicketsPage(query, { 
     routeKey: "customer-dashboard-list" 
   });
   
   // ❌ RUIM: Genérico
   await ZammadGateway.searchOperationalTicketsPage(query, { 
     routeKey: "unknown" 
   });
   ```

### Escrita (mutações)

1. **Camada de `application`** — Toda mutação passa por use case
   ```typescript
   // ✅ BOM: Use case centraliza lógica
   export async function addTicketReply(ticketId: string, body: string) {
     // Validações, audit, etc
     return ZammadGateway.addTicketReply(ticketId, body);
   }
   
   // ❌ RUIM: Chamar gateway diretamente em componente ou rota
   // Sem validação, sem auditoria, sem tratamento de erro coerente
   ```

2. **Isolamento em infraestrutura** — Gateway fica em `infrastructure`, não em `application`
3. **Side effects centralizados** — Cache invalidation, logging estruturado e observabilidade acontecem no gateway

### Contingência e degradação

1. **Mensagens claras ao usuário** quando dados vêm de contingência
   ```typescript
   // UI precisa indicar: "Dados podem estar desatualizados"
   if (routeHealth.stale) {
     showWarning(`Dados atualizados há ${routeHealth.staleMinutes} minutos`);
   }
   ```

2. **Dashboard sem Zammad** — Listagens devem usar cache local mesmo se offline
3. **Health checks públicos** — Expor saúde das rotas via endpoint para monitoramento externo

## Observabilidade

### Métricas estruturadas

Toda requisição ao Zammad é registrada estruturadamente:

```json
{
  "type": "zammad_request",
  "ts": "2026-03-28T14:30:45.123Z",
  "routeKey": "customer-dashboard-list",
  "endpoint": "ticket_customers?state=open",
  "statusCode": 200,
  "ok": true,
  "timeout": false,
  "attempts": 1,
  "latencyMs": 245
}
```

### Health por rota

Rastrear saúde de cada `routeKey`:

```typescript
interface RouteHealth {
  routeKey: string;
  stale: boolean;
  staleMinutes: number;
  staleSince: string | null;
  updatedAt: string;
}
```

### Snapshots de tráfego

Visualizar padrões em janelas de tempo:

```typescript
interface RouteSnapshot {
  routeKey: string;
  total: number;        // requisições no período
  errors: number;       // requisições com erro
  timeouts: number;     // requisições que timeout-aram
  avgLatencyMs: number; // média de latência
  errorRate: number;    // percentual
  timeoutRate: number;  // percentual
}
```

## O que não devemos retomar

- Proposta de usar o Zammad como provedor OAuth principal do portal
- Texto normativo baseado em `NextAuth.js` como diretriz oficial
- Página ou componente consultando Zammad diretamente (sem adapter)
- Várias consultas independentes para alimentar a mesma tela
- Assumir que Zammad sempre está disponível (sem fallback)

## Quando abrir um novo recorte de integração

Abrir documentação técnica **nova** apenas quando houver um destes casos:

- Novo webhook ou mudança estrutural em handlers existentes
- Novo fluxo de mutação relevante (criar ticket, atualizar campo personalizado, etc)
- Nova estratégia de sincronização ou cache
- Integração bidirecional com outro módulo (ex: remoto ↔ tickets)

Quando isso acontecer, o documento **deve registrar:**

1. **Contrato de entrada e saída**
   - Qual dado entra e sai de qual endpoint
   - Tipos esperados (Zod schemas recomendado)

2. **Fallback esperado**
   - O que acontece se Zammad falha
   - Como cache local mitiga falhas

3. **Impacto em cache local**
   - Qual resultado é cacheado
   - Por quanto tempo (TTL)
   - Estratégia de invalidação

4. **Observabilidade mínima**
   - Qual `routeKey` usar
   - Métricas críticas a monitorar
   - Alertas esperados

## Integração com módulo remoto

O Zammad integra bidirecionalmente com o módulo remoto via webhooks:

### De Zammad para Remoto

**Webhook:** `POST /api/integrations/zammad/webhook`

Quando um ticket é criado com tag `#remoto`, o webhook:

1. Resolve a empresa via email do cliente
2. Busca o host remoto correspondente
3. Cria uma sessão de acesso remoto autenticada
4. Vincula ticket ↔ sessão para deep-linking

**Campos esperados no ticket:**
- `customer.email` — para resolver empresa
- Tag `#remoto` — para sinalizar integração
- Custom field com RustDesk ID (opcional)

### De Remoto para Zammad

**Use case:** `resolveRustdeskDeepLink`

Ao clicar em "Ver no Zammad" dentro de uma sessão remota:

1. Resolve ticket a partir de sessão
2. Valida acesso do usuário ao ticket
3. Retorna deep-link para abrir no browser

Arquivo: [`remote/application/zammad-integration.ts`]

## Registro histórico

Houve uma análise inicial propondo:

- SSO com Zammad
- Área "Meus chamados" acoplada ao token do help desk
- Uso de biblioteca de auth como caminho principal da integração

Esse recorte foi mantido apenas como contexto histórico e **não representa mais a diretriz vigente** da plataforma. A integração hoje é isolada, resiliente e opera sob contrato claro com a feature de tickets.
