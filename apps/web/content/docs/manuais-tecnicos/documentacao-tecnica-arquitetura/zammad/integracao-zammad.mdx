## Objetivo

Documentar como o projeto usa o Zammad hoje — arquitetura de cache em banco, sistema de filas, SLA, configurações globais por perfil e diretrizes para novas implementações.

<Callout type="warn" title="Documento atualizado em 2026-04-03">
  Esta versão corrige divergências entre a documentação anterior e o código em produção. A principal correção é que o cache de tickets **não é em memória** — é persistido em PostgreSQL via `ZammadTicketCache`.
</Callout>

---

## Papel atual do Zammad

O Zammad é tratado como:

- **Sistema de atendimento e ticketing** — fonte de verdade para tickets e histórico de interações
- **Provider de eventos** — webhooks para sessões remotas (deep-link, criação de sessões)
- **Origem de observabilidade** — saúde das rotas e contadores de SLA rastreados por rota
- **Adapter consultado** — via gateway dedicado na camada de infraestrutura, nunca diretamente de componentes

O Zammad **não** é tratado como:

- Fonte principal de autenticação do portal
- Backend central da aplicação
- Repositório de regras de negócio da plataforma
- Componente consultado diretamente (sem adapter)

## Cache de tickets em banco (ZammadTicketCache)

<Callout type="info">
  Esta é a principal diferença em relação à documentação anterior. O cache não é em memória — é um modelo Prisma em PostgreSQL com índices otimizados.
</Callout>

### Modelo

```prisma
model ZammadTicketCache {
  id             String    @id @default(cuid())
  zammadTicketId Int       @unique
  number         String
  title          String
  state          String?
  stateId        Int?
  priorityId     Int?
  groupName      String?
  customer       String?
  ownerId        Int?
  firstResponseAt DateTime?
  resolvedAt     DateTime?
  escalationAt   DateTime?
  breached       Boolean   @default(false)
  createdAtZammad DateTime
  updatedAtZammad DateTime
  lastEventType  String?
  lastSyncedAt   DateTime  @default(now())
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt
}
```

### Por que banco e não memória?

| Critério | In-memory | PostgreSQL (atual) |
|---|---|---|
| Sobrevive a restart | ❌ | ✅ |
| Compartilhado entre instâncias | ❌ | ✅ |
| Consultável por múltiplas rotas | Limitado | ✅ |
| Fallback em degradação | Perdido no restart | Persiste |
| Indexação por stateId, ownerId | Manual | Via `@@index` nativo |

### Fluxo de atualização do cache

```
Zammad → Webhook → /api/integrations/zammad/webhook
                          ↓
                   ZammadTicketCache.upsert()
                   ZammadSyncState.update(lastWebhookAt)

Worker periódico → POST /api/platform/zammad/sync
                          ↓
                   Busca tickets na API Zammad
                   ZammadTicketCache.upsert() em lote
                   ZammadSyncState.update(lastWorkerSyncAt)
```

### staleWarning

Quando `ZammadSyncState.lastWorkerSyncAt` ou `lastWebhookAt` ultrapassam o threshold configurado, `getTicketsAction` retorna um `staleWarning` que é exibido no `TicketsContainer`:

```typescript
// TicketsContainer.tsx
{staleWarning && (
  <Alert className="border-amber-500/40 bg-amber-500/10">
    <AlertTitle>Dados em modo contingência</AlertTitle>
    <AlertDescription>{staleWarning}</AlertDescription>
  </Alert>
)}
```

## Sistema de filas (QueueKey)

A lista de tickets suporta filtros de fila em tempo real, todos resolvidos contra o `ZammadTicketCache`:

```typescript
export type QueueKey = "all" | "my_queue" | "unassigned" | "critical" | "no_response";
```

| Fila | Lógica de filtragem |
|---|---|
| `all` | Sem filtro adicional |
| `my_queue` | `ownerId` do usuário logado no Zammad |
| `unassigned` | `ownerId IS NULL` |
| `critical` | `priorityId = 3` (Alta) |
| `no_response` | `firstResponseAt IS NULL` AND estado operacional |

Cada fila retorna seu contador em `queueCounts` para exibição nos botões de filtro do `TicketsContainer`.

## Grupos de status (TicketStatusGroup)

```typescript
export type TicketStatusGroup = "open" | "pending" | "closed";
```

O mapeamento `stateId → grupo` é configurável via `ZAMMAD_STATE_MATRIX_JSON` (variável de ambiente). O padrão (hardcoded em `packages/core`) é:

```json
{
  "statusByStateId": {
    "1": "Aberto",
    "2": "Em Análise",
    "3": "Pendente",
    "4": "Resolvido",
    "5": "Resolvido",
    "6": "Em Análise",
    "7": "Pendente"
  }
}
```

### Janela de tickets fechados (closedWindow)

Para o grupo `closed`, o usuário pode filtrar por janela de tempo:

```typescript
export type ClosedTicketsWindow = "30d" | "60d" | "90d" | "180d" | "365d" | "all";
```

## SLA — cálculo e rastreamento

O SLA é calculado client-side via `computeTicketSla` do pacote `@dosc-syspro/core`:

```typescript
// packages/core/src/services/zammad-sla.ts
export function computeTicketSla(input: {
  createdAt: Date;
  firstResponseAt?: Date | null;
  resolvedAt?: Date | null;
  priorityId?: number | null;
  now?: Date;
}): TicketSlaMeta {
  // Priority 3 (Alta) → SLA × 0.5 (mais apertado)
  // Priority 1 (Baixa) → SLA × 2 (mais folgado)
  // Priority 2 (Normal) → SLA base (padrão: 120 min)
}
```

Variáveis de ambiente que controlam o SLA:

```bash
ZAMMAD_SLA_FIRST_RESPONSE_MINUTES=120  # padrão: 120 min
ZAMMAD_SLA_WARNING_MINUTES=15          # alerta: 15 min antes do breach
```

O campo `ZammadTicketCache.breached` é persistido no banco para consultas rápidas nas filas críticas.

### Exibição na tabela

```
SlaBadge:
  breached → Badge destructive "SLA estourado"
  warning  → Badge amber "SLA alerta (~X min)"
  normal   → Badge secondary "No prazo"
```

## Configurações globais por perfil (ZammadGlobalSettings)

Um conjunto de configurações governa como o portal cria tickets no Zammad para cada perfil de usuário:

```typescript
// packages/contracts/src/zammad-global-settings.ts
export type ZammadGlobalSettings = {
  defaultGroup: string;
  defaultPriorityId: number;
  defaultStateId: number;
  defaultArticleType: "note" | "phone" | "email";
  defaultArticleInternal: boolean;
  defaultOwnerMode: "UNASSIGNED" | "ASSIGN_CURRENT_AGENT";
  titlePrefix: string;            // ex: "[Portal]"
  roleDefaults: {
    clienteAdmin: { group, stateId, ownerMode, priorityId };
    clienteUser:  { group, stateId, ownerMode, priorityId };
    admin:        { group, stateId, ownerMode, priorityId };
    suporte:      { group, stateId, ownerMode, priorityId };
    developer:    { group, stateId, ownerMode, priorityId };
  };
};
```

Gerenciadas em **Configurações → Zammad** pelo formulário `ZammadGlobalSettingsForm`. O catálogo de grupos, estados e prioridades é carregado ao vivo do Zammad com fallback em snapshot.

## Chat e abertura de chamados

### TicketChat

O chat de tickets utiliza `ReactQuill` (SSR desabilitado via `dynamic import`) para composição de respostas ricas:

```typescript
const ReactQuill = dynamic(() => import("react-quill-new"), { ssr: false });
```

Funcionalidades:
- Formatação básica (bold, italic, underline, listas)
- Upload de arquivos via `<input type="file" multiple>`
- Preview de arquivos anexados com remoção individual
- Detecção de ticket fechado (bloqueia envio)
- Auto-scroll para última mensagem via `scrollRef`

### TicketSheet (abertura)

Formulário lateral (`Sheet`) para abertura de novos tickets. Para usuários sistema (`isSystemUser`), apresenta um picker de cliente com busca por nome/empresa.

## Quick Actions

Ações rápidas disponíveis para administradores diretamente na tabela de tickets e na tela de detalhe:

```typescript
type QuickAction = "assume" | "priority_high" | "macro_followup";

// Endpoint:
POST /api/platform/tickets/:ticketId/quick-actions
{ action: "assume" | "priority_high" | "macro_followup" }
```

| Ação | Efeito no Zammad |
|---|---|
| `assume` | Atribui `ownerId` ao usuário logado |
| `priority_high` | Altera prioridade para 3 (Alta) |
| `macro_followup` | Aplica macro de follow-up configurada |

## Integração com módulo remoto

O Zammad integra bidirecionalmente com o módulo de acesso remoto:

### Webhook → RemoteSession

Quando ticket com tag `#remoto` é criado:
1. Webhook recebe evento
2. Resolve empresa via `customer.email`
3. Busca `RemoteHost` correspondente
4. Cria `RemoteSession` com `ticketId` vinculado

### RemoteSession → Zammad

Ao iniciar/encerrar sessão remota, nota interna é adicionada ao ticket:
```
POST /ticket_articles
{ ticket_id, body: "<p>Sessão remota iniciada...</p>", internal: true }
```

Configurado via `configureRemoteSessionTicketNoteHandler` no `packages/api`.

## Variáveis de ambiente

```bash
# Conectividade
ZAMMAD_URL=https://zammad.example.com

# Cache e fallback
ZAMMAD_FALLBACK_MAX_STALE_MINUTES=15

# Sync worker
ZAMMAD_SYNC_PAGES=3
ZAMMAD_SYNC_PER_PAGE=50

# SLA
ZAMMAD_SLA_FIRST_RESPONSE_MINUTES=120
ZAMMAD_SLA_WARNING_MINUTES=15

# State matrix (opcional — sobrescreve padrão do core)
ZAMMAD_STATE_MATRIX_JSON='{"statusByStateId":{"1":"Aberto",...}}'
```

## O que não retomar

- SSO com Zammad como provider OAuth principal
- Cache em memória puro (sem persistência em banco)
- Componente ou página consultando a API Zammad diretamente
- Múltiplas consultas independentes para a mesma tela
- Assumir que Zammad sempre está disponível
