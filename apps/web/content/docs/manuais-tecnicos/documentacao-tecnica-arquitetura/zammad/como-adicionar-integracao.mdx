## Checklist antes de começar

- [ ] **O dado precisa vir do Zammad ao vivo** ou pode ser servido do `ZammadTicketCache`?
- [ ] Qual endpoint Zammad usar?
- [ ] A operação é leitura (GET) ou escrita (POST/PUT)?
- [ ] RouteKey semântico definido?
- [ ] Como o dado afeta o `ZammadTicketCache` (upsert, invalidação)?
- [ ] A UI precisa de feedback de staleness?

<Callout type="info">
  Para a maioria das leituras de tickets, o `ZammadTicketCache` já tem o que você precisa. Prefira consultar o cache antes de criar uma nova rota para o Zammad.
</Callout>

## Caso 1: Filtrar tickets por campo customizado

### Contexto

Você precisa listar tickets de um grupo específico ou com um campo custom. O `ZammadTicketCache` já tem `groupName`, `stateId`, `priorityId`, `ownerId`. Se o campo está lá, **use o cache** — sem nova consulta ao Zammad.

### Passo 1: Verificar se o dado já está no cache

```typescript
// ✅ Use ZammadTicketCache para campos já presentes
const tickets = await prisma.zammadTicketCache.findMany({
  where: {
    groupName: "Suporte Nível 2",
    stateId: { in: OPEN_STATE_IDS },
  },
  orderBy: [{ updatedAtZammad: "desc" }],
  take: 50,
});
```

Se o campo **não está no cache**, vá para o Passo 2.

### Passo 2: Adicionar campo ao ZammadTicketCache (se necessário)

Crie uma migration:

```sql
-- packages/database/prisma/migrations/YYYYMMDD_add_ticket_cache_field.sql
ALTER TABLE "ZammadTicketCache"
ADD COLUMN "customField" TEXT;

CREATE INDEX "ZammadTicketCache_customField_idx"
ON "ZammadTicketCache"("customField");
```

Atualize o schema Prisma:

```prisma
model ZammadTicketCache {
  // ...campos existentes...
  customField String?

  @@index([customField])
}
```

Atualize o worker de sync para popular o novo campo via API Zammad.

### Passo 3: Adicionar à action existente

Se é uma nova variante de filtro, adicione ao `getTicketsAction`:

```typescript
// features/tickets/application/actions.ts
export async function getTicketsAction(params: TicketQueryParams) {
  const where: Prisma.ZammadTicketCacheWhereInput = {
    // ...filtros existentes...
    ...(params.customField && { customField: params.customField }),
  };

  const [tickets, total] = await Promise.all([
    prisma.zammadTicketCache.findMany({ where, ...paginacao }),
    prisma.zammadTicketCache.count({ where }),
  ]);

  return { data: mapTickets(tickets), pagination: buildPagination(total, params) };
}
```

## Caso 2: Nova leitura direta no Zammad (dado não cacheável)

### Contexto

Você precisa de um dado que não faz sentido manter no cache (ex: histórico de auditoria, relatório ad-hoc).

### Passo 1: Interface no domain

```typescript
// features/tickets/domain/repositories/zammad-gateway.repository.ts
export interface ZammadGatewayRepository {
  // ...existentes...

  // ✅ NOVO
  getTicketAuditTrail(
    ticketId: string | number,
    options?: { routeKey?: string }
  ): Promise<ZammadAuditEntry[]>;
}
```

### Passo 2: Implementação no gateway

```typescript
// features/tickets/infrastructure/gateways/zammad-gateway.ts
async getTicketAuditTrail(ticketId, options) {
  const routeKey = options?.routeKey ?? "get-ticket-audit";

  const data = await fetchZammad(
    `ticket_history/${ticketId}`,
    { cache: "no-store" },
    { routeKey, cacheTtlSeconds: 0 }  // ← audit: nunca cachear
  );

  return (data?.history ?? []).map(entry => ({
    id: entry.id,
    type: entry.type,
    createdAt: entry.created_at,
    userId: entry.created_by_id,
  }));
}
```

### Passo 3: Use case no application

```typescript
// features/tickets/application/use-cases.ts
export async function getTicketAuditTrail(ticketId: string) {
  if (!ticketId?.trim()) {
    throw new Error("Ticket ID obrigatório");
  }

  try {
    return await ZammadGateway.getTicketAuditTrail(ticketId, {
      routeKey: "ticket-audit-trail",
    });
  } catch (error) {
    logger.error("Falha ao carregar audit trail", { ticketId, error });
    throw error; // propagar — não mascarar
  }
}
```

### Passo 4: API route

```typescript
// app/api/platform/tickets/[id]/audit/route.ts
export async function GET(
  _request: Request,
  { params }: { params: Promise<{ id: string }> }

    return NextResponse.json({ error: "Falha ao carregar audit" }, { status: 500 });
  }
}
```

## Caso 3: Nova escrita (mutação) no Zammad

### Contexto

Você precisa atualizar um campo de ticket ou adicionar um artigo interno. Mutações sempre passam pelo use case — nunca direto do componente.

### Padrão para mutações

```typescript
// ✅ BOM: use case centraliza validação, auditoria e side-effects
export async function updateTicketPriority(
  ticketId: string,
  priorityId: 1 | 2 | 3,
  actorUserId: string
) {
  // 1. Validação
  if (!ticketId || !priorityId) {
    throw new Error("ticketId e priorityId são obrigatórios");
  }

  // 2. Verificar acesso (se necessário)
  const hasAccess = await canUserAccessTicket(ticketId, actorUserId);
  if (!hasAccess) throw new Error("Sem acesso ao ticket");

  // 3. Chamar gateway (escrita nunca usa cache)
  try {
    const result = await ZammadGateway.updateTicketPriority(ticketId, priorityId);

    // 4. Atualizar ZammadTicketCache (consistência local imediata)
    await prisma.zammadTicketCache.updateMany({
      where: { zammadTicketId: Number(ticketId) },
      data: { priorityId, updatedAtZammad: new Date() },
    });

    // 5. Log estruturado
    logger.info("Prioridade alterada", {
      ticketId,
      priorityId,
      actorUserId,
    });

    return result;
  } catch (error) {
    logger.error("Falha ao alterar prioridade", { ticketId, priorityId, error });
    throw error;
  }
}
```

<Callout type="warn">
  Após qualquer mutação que altere campos presentes no `ZammadTicketCache`, atualize o cache imediatamente com `prisma.zammadTicketCache.updateMany`. Isso evita inconsistência até o próximo sync do worker.
</Callout>

### Implementação no gateway para escrita

```typescript
async updateTicketPriority(
  ticketId: string | number,
  priorityId: number
): Promise<unknown> {
  return fetchZammad(`tickets/${ticketId}`, {
    method: "PUT",
    body: JSON.stringify({ priority_id: priorityId }),
    cache: "no-store",  // ← escritas NUNCA usam cache
    headers: { "Content-Type": "application/json" },
  });
}
```

## Caso 6: Novo webhook do Zammad

### Contexto

Você precisa reagir a um novo evento do Zammad (ex: ticket escalado, SLA prestes a estourar).

### Passo 1: Registrar no Zammad

No painel Zammad → WebHooks → New:
- URL: `https://seu-dominio/api/integrations/zammad/webhook`
- Eventos: selecionar o evento desejado
- Token de autenticação compartilhado

### Passo 2: Handler

```typescript
// app/api/integrations/zammad/webhook/route.ts
export async function POST(request: Request) {
  // 1. Validar autenticação
  const isValid = isValidHmacSignature(
    await request.text(),
    request.headers.get("X-Hub-Signature"),
    process.env.ZAMMAD_WEBHOOK_SECRET!
  );

  if (!isValid) {
    return NextResponse.json({ error: "Não autorizado" }, { status: 401 });
  }

  const payload = await request.json();

  // 2. Identificar evento
  const ticketId = payload.ticket?.id;
  const stateId = payload.ticket?.state_id;
  const eventType = payload.type ?? "unknown";

  if (!ticketId) {
    return NextResponse.json({ error: "Payload inválido" }, { status: 400 });
  }

  try {
    // 3. Atualizar ZammadTicketCache (sempre)
    await prisma.zammadTicketCache.upsert({
      where: { zammadTicketId: ticketId },
      update: {
        state: payload.ticket?.state,
        stateId,
        lastEventType: eventType,
        lastSyncedAt: new Date(),
      },
      create: buildTicketCacheFromWebhookPayload(payload.ticket),
    });

    // 4. Atualizar ZammadSyncState
    await prisma.zammadSyncState.upsert({
      where: { key: "global" },
      update: { lastWebhookAt: new Date() },
      create: { key: "global", lastWebhookAt: new Date(), updatedAt: new Date() },
    });

    // 5. Side-effects específicos do evento
    await handleZammadTicketEvent(ticketId, eventType, payload);

    logger.info("Webhook processado", { ticketId, eventType });

    // ⚠️ Retornar 200 mesmo com erros parciais — Zammad não deve reenviar
    return NextResponse.json({ success: true });
  } catch (error) {
    logger.error("Erro ao processar webhook", { error, ticketId, eventType });
    return NextResponse.json({ success: false, error: String(error) });
  }
}

async function handleZammadTicketEvent(
  ticketId: number,
  eventType: string,
  payload: unknown
) {
  // Dispatcher de eventos — adicionar novos handlers aqui
  if (eventType === "ticket.escalation") {
    await handleTicketEscalation(ticketId, payload);
  }

  if (eventType === "ticket.close") {
    // Verificar sessão remota ativa
    const session = await prisma.remoteSession.findFirst({
      where: { ticketId: String(ticketId), status: { in: ["REQUESTED", "STARTED"] } },
    });

    if (session) {
      await prisma.remoteSession.update({
        where: { id: session.id },
        data: { status: "ENDED", endedAt: new Date() },
      });
    }
  }
}
```

<Callout type="warn">
  Sempre valide a autenticação do webhook. Use `isValidHmacSignature` do `@dosc-syspro/api/request-auth`. Qualquer pessoa pode chamar seu endpoint sem validação.
</Callout>

## Troubleshooting

### staleWarning aparece mesmo com Zammad online

→ Worker de sync não rodou no período esperado. Verificar cron/scheduler.
→ `ZammadSyncState.lastWorkerSyncAt` está desatualizado? Rodar manualmente:
```bash
curl -X POST /api/platform/zammad/sync \
  -H "Authorization: Bearer $ZAMMAD_SYNC_TOKEN"
```

### Cache com dados desatualizados após mutação

→ Use case não atualizou `ZammadTicketCache` após escrita no Zammad.
→ Adicionar `prisma.zammadTicketCache.updateMany` após mutação bem-sucedida.

### "Circuit breaker ativo para rota Zammad"

→ Rota falhou 3x. Aguardar 20s (COOLDOWN) e tentar novamente.
→ Se persistir, verificar logs do Zammad e status da API.

### Webhook não dispara

1. Verificar URL configurada no Zammad (typo?)
2. Verificar token de autenticação (ZAMMAD_WEBHOOK_SECRET)
3. Ver logs do Zammad → WebHooks → histórico de envios
4. Seu endpoint retornou 200? Se 4xx/5xx, Zammad reenviará

### "AGENT_TOKEN" nos logs do webhook

→ Confusão entre webhook do Zammad e webhook do módulo remoto. São endpoints diferentes:
- Zammad: `POST /api/integrations/zammad/webhook`
- Remote (Discover): `POST /api/remote/rustdesk/discover`
