### Passo 2: Implementar na gateway

Arquivo: `features/tickets/infrastructure/gateways/zammad-gateway.ts`

```typescript
export const ZammadGateway: ZammadGatewayRepository = {
  // ... existentes ...
  
  // ✅ NOVO
  async getTicketsByCustomerEmail(
    email: string,
    options?: {
      limit?: number;
      page?: number;
      cacheTtlSeconds?: number;
      routeKey?: string;
    }
  ): Promise<ZammadOperationalTicket[]> {
    const limit = options?.limit ?? 100;
    const page = options?.page ?? 1;
    const routeKey = options?.routeKey ?? "get-tickets-by-email";

    // 1. Construir query (email e estado operacional)
    const query = buildCustomerEmailsQuery([email]);
    const stateQuery = buildStateQuery([OPEN, IN_PROGRESS, PENDING]);
    const fullQuery = `${query} ${stateQuery}`;

    // 2. Buscar com cache
    const data = await fetchZammad(
      `tickets/search?query=${encodeURIComponent(fullQuery)}&limit=${limit}&page=${page}`,
      { cache: "no-store" },
      { 
        routeKey,
        cacheTtlSeconds: options?.cacheTtlSeconds ?? 60 // 1 min padrão
      }
    );

    // 3. Normalizar resposta
    const tickets = normalizeSearchResponse(data);
    const parsed = tickets.map(t => zammadOperationalTicketSchema.parse(t));
    
    return parsed;
  }
};
```

### Passo 3: Usar na application

Arquivo: `features/tickets/application/use-cases.ts`

```typescript
// ✅ NOVO
export async function getCustomerTickets(
  customerEmail: string,
  options?: { page?: number }
) {
  // Validar email
  if (!customerEmail || !customerEmail.includes("@")) {
    throw new Error("Email inválido");
  }

  // Chamar gateway (com fallback automático)
  try {
    return await ZammadGateway.getTicketsByCustomerEmail(
      customerEmail,
      {
        limit: 50,
        page: options?.page ?? 1,
        cacheTtlSeconds: 120,
        routeKey: "customer-tickets-page"
      }
    );
  } catch (error) {
    logger.error("Falha ao carregar tickets do cliente", { 
      customerEmail, 
      error 
    });
    throw error; // Propagar para não mascarar erro
  }
}
```

### Passo 4: Expor em API route

Arquivo: `app/api/tickets/customer/route.ts`

```typescript
export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const email = searchParams.get("email");
  const page = searchParams.get("page");

  if (!email) {
    return NextResponse.json(
      { error: "Email requerido" },
      { status: 400 }
    );
  }

  try {
    const tickets = await getCustomerTickets(email, {
      page: page ? parseInt(page) : 1
    });

    return NextResponse.json({ tickets });
  } catch (error) {
    logger.error("GET /api/tickets/customer falhou", { email, error });
    return NextResponse.json(
      { error: "Falha ao carregar tickets" },
      { status: 500 }
    );
  }
}
```

```typescript
  ticketId: string | number, 
  body: string
): Promise<unknown> {

    method: "POST",
```

## Caso 3: Implementar novo webhook

### Contexto

Zammad te notifica quando um ticket entra em estado "fechado". Você precisa reagir (ex: limpar cache, notificar Slack, fechar sessão remota).

### Passo 1: Registrar webhook no Zammad

No painel Zammad:
- WebHooks → New
- URL: `https://seu-dominio/api/integrations/zammad/webhook`
- Eventos: `ticket.state.change` onde `state = 4` (closed)
- Autenticação: Token compartilhado com seu backend

### Passo 2: Implementar handler

Arquivo: `app/api/platform/zammad/webhook/route.ts`

```typescript
export async function POST(request: Request) {
  // 1. Verificar autenticação
  const token = request.headers.get("Authorization");
  if (token !== `Bearer ${ZAMMAD_WEBHOOK_SECRET}`) {
    return NextResponse.json(
      { error: "Não autorizado" },
      { status: 401 }
    );
  }

  // 2. Parser payload
  const payload = await request.json();
  
  // 3. Validar estrutura
  if (!payload.ticket?.id || !payload.ticket?.state_id) {
    return NextResponse.json(
      { error: "Payload inválido" },
      { status: 400 }
    );
  }

  try {
    // 4. Executar lógica
    const handled = await handleZammadTicketStateChange(
      payload.ticket.id,
      payload.ticket.state_id
    );

    // 5. Log estruturado
    logger.info("Webhook processado", {
      type: "zammad_webhook",
      ticketId: payload.ticket.id,
      handled
    });

    // 6. Confirmar ao Zammad
    return NextResponse.json({ success: true });
  } catch (error) {
    logger.error("Erro ao processar webhook", { error });
    // ⚠️ Retornar 200 mesmo com erro para Zammad não reenviar indefinidamente
    return NextResponse.json({ success: false, error: String(error) });
  }
}

async function handleZammadTicketStateChange(
  ticketId: string,
  stateId: number
) {
  const CLOSED_STATE_ID = 4;

  if (stateId === CLOSED_STATE_ID) {
    // Buscar sessão remota associada
    const session = await prisma.remoteSession.findFirst({
      where: { ticketId }
    });

    if (session) {
      // Fechar sessão
      await prisma.remoteSession.update({
        where: { id: session.id },
        data: { status: "CLOSED", closedAt: new Date() }
      });

      // Invalidar cache do ticket
      // (implementar conforme necessário)

      return true;
    }
  }

  return false;
}
```

## Troubleshooting

### "Circuit breaker ativo para rota Zammad"

→ Sua rota falhou 3 vezes. Aguarde 20 segundos e tente novamente.
→ Se continuar, debugar no Zammad (logs, performance, disponibilidade).

### Dados retornam como "stale"

→ Cache tem > X minutos. Normal. UI deve avisar o usuário.
→ Se stale > 15 minutos, houve indisponibilidade longa. Debugar Zammad.

### "Timeout após tentativas"

→ Zammad respondendo lentamente. Verificar logs do Zammad ou rede.
→ Se consistente, aumentar `ZAMMAD_TIMEOUT_MS` é solução de curto prazo. Debugar causa real.

### Webhook não dispara

1. Verificar URL no Zammad (typo?)
2. Verificar token de autenticação
3. Verificar logs do Zammad (section WebHook)
4. Zammad enviou POST? Se sim, checar status na response
5. Seu endpoint retornou 200? Se não, Zammad reenviará
