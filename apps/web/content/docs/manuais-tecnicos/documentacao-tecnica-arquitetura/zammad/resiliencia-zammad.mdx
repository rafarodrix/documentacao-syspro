## Contexto

O Zammad é um sistema externo gerenciado por terceiros. Pode ficar indisponível, lento ou instável. A arquitetura de integração foi desenhada para **nunca falhar silenciosamente** nem deixar o usuário sem resposta.

<Callout type="warn" title="Correção em relação à versão anterior">
  O cache de contingência **não é em memória** — é o modelo `ZammadTicketCache` em PostgreSQL. Isso garante que dados sobrevivam a restarts, sejam compartilhados entre instâncias e estejam sempre disponíveis para consulta, mesmo que o Zammad esteja completamente offline.
</Callout>

---

## Visão geral das camadas

```
Requisição do usuário
       ↓
  Cache PostgreSQL (ZammadTicketCache)
       ↓ [stale? → staleWarning no UI]
  Consulta Zammad (se necessário)
       ↓
  Retry (3x, exponential backoff)
       ↓
  Circuit Breaker por rota (opcional)
       ↓
  Fallback: servir ZammadTicketCache existente + staleWarning
       ↓
  Erro controlado (UI notifica, não quebra)
```

A estratégia central é **cache-first**: a UI sempre tem dados do banco para mostrar. A consulta ao Zammad é para atualizar o cache, não para bloquear a renderização.

---

## 1. Cache em banco (ZammadTicketCache) — camada principal

### Por que é a camada mais importante

Ao contrário do cache em memória (perdido em restart), o `ZammadTicketCache` no PostgreSQL é:

- **Persistente** — sobrevive a deploys e restarts
- **Compartilhado** — todas as instâncias do Next.js acessam o mesmo dado
- **Consultável** — suporta filtros complexos (por stateId, ownerId, priorityId, groupName, breach)
- **Indexado** — índices em `updatedAtZammad`, `stateId + priorityId`, `ownerId`, `breached`

### Atualização do cache

O cache é atualizado por dois mecanismos independentes:

**1. Webhooks (tempo real)**

```
Zammad emite evento
      ↓
POST /api/integrations/zammad/webhook
      ↓
prisma.zammadTicketCache.upsert({ where: { zammadTicketId } })
prisma.zammadSyncState.upsert({ key: "global", data: { lastWebhookAt: now } })
```

**2. Worker de sincronização (periódico)**

```
POST /api/platform/zammad/sync (cron a cada ~5 min)
      ↓
Busca N páginas na API Zammad
      ↓
prisma.zammadTicketCache.upsert() em lote
prisma.zammadSyncState.upsert({ lastWorkerSyncAt: now })
```

### staleWarning

Quando nenhum dos mecanismos rodou recentemente, `getTicketsAction` detecta e sinaliza:

```typescript
// Lógica em getTicketsAction
const syncState = await prisma.zammadSyncState.findUnique({ where: { key: "global" } });

const lastSync = syncState?.lastWebhookAt ?? syncState?.lastWorkerSyncAt;
const minutesSinceSync = lastSync
  ? (Date.now() - lastSync.getTime()) / 60000
  : Infinity;

const staleWarning = minutesSinceSync > ZAMMAD_FALLBACK_MAX_STALE_MINUTES
  ? `Dados com ${Math.round(minutesSinceSync)} minutos. Zammad pode estar indisponível.`
  : undefined;
```

No `TicketsContainer`, o aviso é exibido em faixa amarela sem bloquear a listagem.

## 2. Retry com exponential backoff

### Por que

Falhas transitórias (network glitch, GC pause no Zammad) devem ser recuperadas automaticamente. Retry imediato não dá tempo para resolução.

### Configuração

```bash
ZAMMAD_RETRY_MAX_ATTEMPTS=3       # padrão: 3
ZAMMAD_RETRY_BASE_DELAY_MS=400    # padrão: 400ms
```

### Sequência de delays

```
Tentativa 1: imediato
Tentativa 2: 400ms
Tentativa 3: 800ms
Total worst case: ~1.2s + tempo de resposta do Zammad
```

### Código (`zammad-http-client.ts`)

```typescript
for (let attempt = 1; attempt <= ZAMMAD_RETRY_MAX_ATTEMPTS; attempt++) {
  try {
    const response = await fetch(input, options);
    if (response.ok) return response;

    // Retryable: 429 (rate limit) e 5xx (erro servidor)
    if (!isRetryableStatus(response.status) || attempt === max) {
      throw new Error(`HTTP ${response.status}`);
    }
  } catch (error) {
    if (attempt === ZAMMAD_RETRY_MAX_ATTEMPTS) throw error;
  }

  const delay = ZAMMAD_RETRY_BASE_DELAY_MS * Math.pow(2, attempt - 1);
  await sleep(delay);
}
```

### Decisão

**Exponential backoff reduz congestionamento.** Se Zammad está lento, bombear com retries imediatos piora. Esperar progressivamente dá tempo para recuperação.

## 3. Circuit breaker por rota (opcional)

### Por que

Se uma rota falha repetidamente, continuar tentando desperdiça tempo de timeout e congestiona o Zammad.

### Ativação

```bash
ZAMMAD_ENABLE_IN_MEMORY_CIRCUIT_BREAKER=true
ZAMMAD_CIRCUIT_COOLDOWN_MS=20000    # cooldown: 20s
```

<Callout type="info">
  O circuit breaker é in-memory por design — é auxiliar ao cache principal do banco. Em cenário de múltiplas instâncias, cada instância tem seu próprio circuit breaker, o que é aceitável porque o cache em banco é o mecanismo principal de resiliência.
</Callout>

### Comportamento

```
Rota falha na última tentativa de retry
      ↓
circuitOpenUntil[routeKey] = Date.now() + COOLDOWN
      ↓
Próximas requisições → cai direto no fallback (ZammadTicketCache)
      ↓
Após 20s → tenta novamente (half-open implícito)
```

### Escopo

O circuit breaker é **por `routeKey`**, não global. Uma rota problemática não bloqueia outras:

```typescript
// ✅ Correto: cada rota tem seu circuit breaker independente
await ZammadGateway.searchOperationalTicketsPage(query, {
  routeKey: "admin-dashboard-list",
});

await ZammadGateway.getTicketById(id, {
  routeKey: "ticket-detail",
});
```

## 4. Timeout

```bash
ZAMMAD_TIMEOUT_MS=10000    # padrão: 10s
```

Implementado via `AbortController`. Em caso de timeout, a requisição cai no mecanismo de retry e, após esgotar tentativas, usa o `ZammadTicketCache`.

**Não aumentar `ZAMMAD_TIMEOUT_MS` para mascarar lentidão do Zammad.** O problema correto a resolver é a performance do Zammad, não o timeout do portal.

## 5. Cache de userIds (em memória, auxiliar)

Para operações que precisam resolver `userId` por email no Zammad frequentemente (ex: mutações que precisam do ID do owner):

```typescript
const ZAMMAD_USER_ID_CACHE_TTL_MS = 300000; // 5 minutos

function getCachedUserId(email: string): number | null | undefined {
  const cached = userIdCache.get(email);
  if (cached && cached.expiresAt > Date.now()) return cached.value;
  userIdCache.delete(email);
  return undefined;
}
```

**5 minutos é suficiente** — emails raramente mudam no Zammad. Cache mais longo poderia servir dados desatualizados após mudanças de conta.

## 6. Worker de sincronização proativa

O endpoint de sync garante que o `ZammadTicketCache` esteja sempre atualizado antes de picos:

```
POST /api/platform/zammad/sync
Authorization: Bearer {ZAMMAD_SYNC_TOKEN}
```

```bash
ZAMMAD_SYNC_PAGES=3       # páginas a sincronizar por ciclo
ZAMMAD_SYNC_PER_PAGE=50   # tickets por página
```

### Características

- **Idempotente** — pode rodar quantas vezes precisar
- **Sem sessão web** — usa Bearer token próprio, não sessão do usuário
- **Upsert seguro** — `WHERE zammadTicketId = ?` evita duplicatas
- **Atualiza ZammadSyncState** — mantém rastreabilidade do último sync

## 7. Cálculo de SLA (camada de domínio)

O SLA não depende de chamada ao Zammad — é calculado na camada de domínio a partir dos dados do `ZammadTicketCache`:

```typescript
// packages/core/src/services/zammad-sla.ts
const sla = computeTicketSla({
  createdAt: ticket.createdAtZammad,
  firstResponseAt: ticket.firstResponseAt,
  resolvedAt: ticket.resolvedAt,
  priorityId: ticket.priorityId,
  now: new Date(),
});

// sla.breached → ticket estourou SLA
// sla.warning  → faltam < 15 min para estouro
// sla.minutesToBreach → minutos restantes
```

O campo `ZammadTicketCache.breached` é atualizado pelo worker/webhook para consultas de fila eficientes.

## Fluxo completo — usuário acessa dashboard de tickets

```
1. Browser → getTicketsAction({ page, queue, statusGroup, search })

2. getTicketsAction:
   a. Consulta ZammadTicketCache no PostgreSQL (sempre disponível)
   b. Verifica ZammadSyncState → calcula staleWarning se necessário
   c. Aplica filtros: queue, statusGroup, closedWindow, search
   d. Retorna { data, pagination, queueCounts, statusCounts, staleWarning }

3. TicketsContainer renderiza com dados do cache
   → Se staleWarning, exibe faixa amarela
   → Dados sempre presentes (nunca tela vazia por indisponibilidade do Zammad)

4. Em background, worker roda a cada ~5 min:
   → Consulta Zammad API com retry + circuit breaker
   → Atualiza ZammadTicketCache em lote
   → Atualiza ZammadSyncState.lastWorkerSyncAt

5. Webhook (se disponível):
   → Atualiza ZammadTicketCache em tempo real
   → Atualiza ZammadSyncState.lastWebhookAt
```

## Observabilidade — alertas recomendados

| Métrica | Threshold | Ação |
|---|---|---|
| `errorRate > 10%` por 5 min | Alerta | Investigar Zammad |
| `staleMinutes > FALLBACK_MAX` | Alerta | Worker não rodou ou Zammad offline |
| `avgLatencyMs > 5000` | Warning | Possível gargalo no Zammad |
| `breachedCount` crescendo | Alerta de negócio | SLA comprometido |
| `noResponseCount > N` | Alerta de negócio | Tickets sem primeira resposta |

## Decisões não implementadas (e por quê)

### Não há Redis para cache de tickets

**Por quê:** O `ZammadTicketCache` em PostgreSQL é suficiente e remove uma dependência de infraestrutura. Redis seria necessário apenas se o volume de consultas exigisse latência < 1ms — o que não é o caso para um portal de atendimento.

### Não há request coalescing

**Por quê:** O cache em banco já resolve o N+1 naturalmente — múltiplos usuários lendo simultaneamente consultam a mesma tabela cacheada, não o Zammad diretamente.

### Não há bulkhead pattern entre rotas

**Por quê:** Circuit breaker por `routeKey` já isola falhas. Bulkhead (thread pools separadas) é prematuro para Next.js single-threaded.
