## 1. Estratégia de Validação de Token

Todo agentToken passa por **3 validações sequenciais** em cada requisição (sync, ack, heartbeat):

### 1.1 Validação 1: Token Existe?

```typescript
// packages/remote-domain/src/use-cases/process-sync.ts
const tokenContext = await deps.port.resolveSyncContextByAgentToken(agentToken);

if (!tokenContext) {
  // Token nunca foi emitido ou foi revogado
  throw new Error("AGENT_TOKEN_INVALID");
}
```

**Falha → Resposta HTTP 401**

Motivos:

- Token foi revogado manualmente via UI portal
- Token nunca foi emitido (agente tentando sync antes de bootstrap)
- Typo no token enviado pelo agente

Recuperação:

- ✅ Agente tenta novo bootstrap com `installToken`
- ❌ Se installToken também inválido: suporte precisa reinstalar agente

---

### 1.2 Validação 2: Token Expirou?

```typescript
if (deps.port.isAgentTokenExpired(tokenContext)) {
  // Token e permanentemente inválido a partir de TOKEN_EXPIRY_DATE
  throw new Error("AGENT_TOKEN_EXPIRED");
}
```

**Falha → Resposta HTTP 401 com error_code: AGENT_TOKEN_EXPIRED**

TTL padrão: **90 dias desde issuance**

Motivos:

- Token foi emitido 90+ dias atrás
- Relógio da máquina pode estar errado (menos comum)

Recuperação automática:

- Agente detecta `AGENT_TOKEN_EXPIRED` em sync
- Agente automaticamente tenta fazer novo bootstrap com `installToken`
- Portal reaplica compliance checks no novo token

Observabilidade:

```json
{
  "event": "remote.domain.sync.failed",
  "hostId": "host-123",
  "error": "AGENT_TOKEN_EXPIRED",
  "agentTokenIssuedAt": "2026-01-01T10:00:00.000Z",
  "expectedExpiry": "2026-03-31T10:00:00.000Z",
  "currentTime": "2026-04-02T15:00:00.000Z",
  "actionRequired": "BOOTSTRAP_NEW_TOKEN"
}
```

---

### 1.3 Validação 3: Contexto Carregável?

```typescript
// Validações de contexto de sync
if (!tokenContext.hostId || !tokenContext.companyId) {
  throw new Error("SYNC_CONTEXT_INVALID");
}
```

**Falha → Resposta HTTP 500**

Motivos:

- Banco de dados corrompido ou indisponível
- Migrações incompletas
- Problema de permissão SQL

Recuperação:

- Portal reenvia resposta `RETRY_LATER` ao agente
- Agente aguarda intervalo exponencial (T, 2T, 4T, 8T...) ate max 1 hora
- DevOps investiga logs de banco de dados

## 2. Transições de Estado e Compliance

### 2.1 Estado PENDING_LINK

Maquina descoberta mas não vinculada a nenhum host:

```
+---discover-+--> PENDING_LINK
             |    (aguardando admin linkar)
             |
         suporte clica "Linkar"
             |
             +--> linked & bootstrap_required
```

Durante PENDING_LINK:

- `allowDiscoveryHeartbeat: true` — agente continua chamando discover a cada 5 min
- Agente nao recebe commands (fila vazia)
- Compliance nao e aplicada (apenas triagem)

Saida do estado:

- **Suporte clica "Linkar Máquina"** → transicao para `linked_host_detected` + `bootstrap_required`
- **Suporte clica "Rejeitar"** → `remoteDiscoveredHost.status = REJECTED`, agente para discover

---

### 2.2 Estado LINKED_HOST_DETECTED

Máquina vinculada e token válido. Fluxo normal:

```
sync ---------> [AGENTE HEARTBEAT OK]
  ↓
  sync commands
  ↓
  ack responses
  ↓
  sync (loop continues)
```

Durante operação normal:

- `lastHeartbeatAt` é atualizado a cada sync
- `sysproUpdates[]` é sincronizado incremental (otimização de payload)
- Comandos são despachados

Saida do estado:

- **Token expira** → agente detecta na próxima sync → novo bootstrap necessário
- **Suporte rotaciona token** → mesma coisa
- **Suporte revoga token** → agente fica offline permanentemente

---

### 2.3 Estado HOST_BOOTSTRAP_REQUIRED

Máquina vinculada mas token inválido/expirado:

```
sync (com token inválido)
  ↓
AGENT_TOKEN_EXPIRED
  ↓
agente chama bootstrap automaticamente
  ↓
novo agentToken emitido
  ↓
volta a LINKED_HOST_DETECTED
```

Durante bootstrap_required:

- `allowDiscoveryHeartbeat: false` — agente para de chamar discover
- `requiresAuthenticatedBootstrap: true` — force bootstrap, não accept discover
- Agente fica offline até receber novo token

---

### 2.4 Compliance Checks Após Bootstrap

Após novo token emitido, **4 validações de configuração**:

```typescript
const compliance = {
  aliasMatch: received_alias == expected_alias,
  versionMatch: rustdesk_version == target_version,
  serverHostMatch: rustdesk_server == config.server,
  apiHostMatch: rustdesk_api == config.api,
  publicKeyMatch: public_key_hash == config.public_key
}
```

Se algum falha:

```json
{
  "event": "remote.domain.bootstrap.compliance_failed",
  "hostId": "host-123",
  "failedChecks": ["versionMatch", "serverHostMatch"],
  "severity": "HIGH",
  "actionRequired": "VERIFY_RUSTDESK_CONFIG"
}
```

Suporte deve:

1. Verificar configurações RustDesk no host (Registry, config file)
2. Forçar reinstalação via NSIS se mismatches críticos
3. Monitorar compliance em dashboard

## 3. Fila de Comandos e Deferral

### 3.1 Despacho de Comandos

Em cada sync response, comandos são inclusos:

```typescript
export type ProcessSyncOutput = {
  contractVersion: "rustdesk.sync.v1",
  syncDirectives: SyncCommandDirective[]  // Comandos a executar
}

export type SyncCommandDirective = {
  id: string,
  type: string,  // Ex: "UPDATE_RUSTDESK", "ROTATE_AGENT_TOKEN", etc
  payload: unknown,
  expiresAt?: string // Validade do comando
}
```

### 3.2 Reconhecimento (ACK)

Agente executa comando e retorna resultado:

```
sync ← syncDirective { id: "cmd-123", type: "UPDATE_RUSTDESK" }
  ↓
agente executa
  ↓
ack → { commandId: "cmd-123", status: "ACKNOWLEDGED", executedAt: "2026-03-28T15:30:00.000Z" }
```

Se comando falhar:

```
ack → { commandId: "cmd-123", status: "FAILED", message: "instalador corrompido", details: {...} }
  ↓
portal loga falha
  ↓
suporte recebe alertas
```

### 3.3 Deferral de Comandos Não Confirmados

Se agente não responde em sync:

```
sync 1: despacha cmd-1, cmd-2
  ↓
cmd-1 reconhecido, cmd-2 nao
  ↓
sync 2: redespacha cmd-2 (deferral)
  ↓
cmd-2 agora reconhecido
```

Máximo de redespatch: **3 vezes** antes de marcar como `TIMEOUT`

Apos timeout:

```json
{
  "event": "remote.domain.command.timeout",
  "commandId": "cmd-2",
  "hostId": "host-123",
  "type": "UPDATE_RUSTDESK",
  "attempts": 3,
  "actionRequired": "FORCE_REBOOT_OR_REINSTALL"
}
```

## 4. Otimização de Payload: Hash-Based Incremental

### 4.1 Problema: Sincronização de Grandes Datasets

Com 100+ máquinas Syspro, transmitir lista completa a cada sync:

- Usa banda desnecessária
- Lentidão em redes precárias
- Overload de processamento

### 4.2 Solução: Hash-Based Incremental Sync

Portal mantém **hash** da última lista enviada:

```typescript
// Algoritmo simplificado
const previousHash = host.lastSysproUpdatesHash;
const currentUpdates = scanDiskForSysproInstallations();
const currentHash = sha256(JSON.stringify(currentUpdates.sorted()));

if (previousHash === currentHash) {
  // Nenhuma mudança, não envia nada
  syncDirectives.push({
    id: "no-op-sync",
    type: "ACKNOWLEDGE_UNCHANGED"
  });
} else {
  // Mudanças detectadas, envia delta
  syncDirectives.push({
    id: "sync-updates",
    type: "UPDATE_SYSPRO_INSTALLATIONS",
    payload: {
      installations: currentUpdates,
      hash: currentHash,
      previousHash
    }
  });
  
  host.lastSysproUpdatesHash = currentHash;
}
```

### 4.3 Daily Snapshot Requirement

**Regra obrigatória**: Ao menos 1 sync por dia deve conter snapshot **completo** (não incremental).

Motivo: Se hash ficar corrompido, ao menos uma vez por dia sistema reconstrói estado correto.

```typescript
const isDailySnapshot = lastFullSnapshot < Date.now() - 24h;

if (isDailySnapshot) {
  // Envia completo, mesmo que hash bate
  response.syncDirectives.push({
    type: "FULL_SNAPSHOT_REQUIRED",
    payload: allInstallations
  });
}
```

Observabilidade:

```json
{
  "event": "remote.domain.syspro_updates.synced",
  "hostId": "host-123",
  "updateCount": 5,
  "payload_bytes": 2048,
  "is_snapshot": false,
  "hash": "abc123...",
  "previousHash": "def456..."
}
```

## 5. Smart Scan para Multi-Disco Syspro

### 5.1 Problema: Redundância em Multi-Disco

Máquinas antigas podem ter Syspro instalado em múltiplos discos (antiga estratégia de failover):

```
C:\ → Syspro v6.5 (ativo)
D:\ → Syspro v6.5 (backup)
E:\ → Syspro v5.0 (deprecated, esquecido)
```

Transmitir tudo aumenta ruído de telemetria.

### 5.2 Solução: Smart Deduplication

Regras de deduplicação:

```typescript
function deduplicateSysproInstallations(discovered: Installation[]): Installation[] {
  const byVersion = groupBy(discovered, 'version');
  
  const deduped = [];
  
  for (const [version, installs] of byVersion) {
    if (installs.length === 1) {
      deduped.push(installs[0]);
    } else {
      // Múltiplas cópias mesma versão
      // Preferir C:\ > D:\ > E:\ > ...
      const preferred = installs.sort((a, b) => 
        a.drive.charCodeAt(0) - b.drive.charCodeAt(0)
      )[0];
      deduped.push(preferred);
    }
  }
  
  return deduped;
}
```

Para evitar scan em toda árvore do disco:

## 7. Estratégia de Recuperação

### 7.1 Falhas que se Auto-Recuperam

✅ Token expirado → aveia novo bootstrap automático

✅ Comando não reconhecido → redespatch até max tentativas

✅ Sync timeout ocasional → próximo sync tenta novamente

✅ Compliance mismatch → monitorar, pode ser legítimo

---

### 7.2 Falhas que Requerem Intervenção

❌ AGENT_TOKEN_INVALID (nunca foi emitido) → novo bootstrap com installToken necessário

❌ SYNC_CONTEXT_INVALID (banco corrompido) → DevOps investiga SQL

❌ Host desapareceu da rede > 24h → suporte verifica máquina

❌ Múltiplos commandos timeout → possível agente parado, reiniciar necessário

---

### 7.3 Runbook para Suporte

| Sintoma | Erro no Log | Ação |
|---------|-------------|------|
| Agente offline > 1h | `HEARTBEAT_TIMEOUT` | Verificar IP, porta, firewall RustDesk |
| Agente parou após bootstrap | `AGENT_TOKEN_INVALID` | Novo bootstrap com installToken |
| Compliance falhou | `compliance_failed: versionMatch=false` | Verificar Registry RustDesk ou reinstalar |
| Commands não executam | `COMMAND_TIMEOUT x3` | Reiniciar agente via RustDesk ou remoto |
| Sync recebe nulo | `SYNC_CONTEXT_INVALID` | DevOps verifica DB, migra schema se necessário |

## Conclusão

Resiliência do módulo remoto é construída em **3 camadas**:

1. **Token Management**: Validação 3-passo + TTL + compliance
2. **State Machine**: Transições bem-definidas com deferral de comandos
3. **Observability**: Eventos estruturados para diagnóstico rápido

Operação requer monitoramento proativo:

- Dashboard daily check
- Logs weekly review
- Prevenção token expirations
- Compliance audit mensal
