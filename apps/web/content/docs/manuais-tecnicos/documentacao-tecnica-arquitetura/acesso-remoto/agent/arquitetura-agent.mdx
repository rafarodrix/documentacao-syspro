## Objetivo

Consolidar a operacao do agente remoto RustDesk e definir a migracao do script monolitico (`trilink-agente.ps1`) para arquitetura modular, sem quebrar o fluxo atual `discover -> bootstrap -> sync -> ack`.

## Estado atual (2026-04-02)

- A modularizacao do agente (`FEAT-002J`) foi concluida e o `trilink-agente.ps1` agora atua como orquestrador de ciclo.
- O contrato de `sync` no dominio remoto ja aceita inventario tecnico expandido.
- A persistencia em `remote_host` ja foi ampliada para os novos blocos de inventario.
- O diretorio e o detalhe de host no portal ja exibem sinais operacionais derivados do inventario.

## Plano de implementacao recomendado (proximas sprints)

### Sprint A (P0): estabilidade operacional do ciclo (concluido em 2026-04-03)

Escopo concluido:

1. orquestracao no `trilink-agente.ps1` ajustada para priorizar `sync` com token valido (estrategia `sync_token_first`) e reduzir bootstrap recorrente.
2. invalidacao de token aplicada apenas apos finalizar a fila de ACK (`pendingTokenInvalidation` + `Decision=rebootstrap_required_after_ack_queue`).

Evidencias no codigo:

- `Decision=sync_token_first` quando ha token local reutilizavel.
- `discover/bootstrap` executados apenas quando nao ha token util ou quando `rebootstrapRequired=true`.
- bootstrap restrito a fluxos explicitos do contrato (`host_bootstrap_required` ou `token_invalid`).
- loop de ACK usa `tokenForAck` estavel para todos os comandos do ciclo e posterga a invalidacao para o fim da fila.

Status atual:

- `UPGRADE_CLIENT` implementado em `application/commands.ps1` com download HTTPS, validacao SHA256 e ACK detalhado.
- portal atualizado para emitir `upgrade_client` somente quando metadados seguros de upgrade estiverem presentes.

Definicao de pronto (atingida):

- comando `UPGRADE_CLIENT` com caminho de sucesso/falha cobertos.
- ciclo nominal de host ativo concluindo sem bootstrap desnecessario.
- token nao e invalidado no meio da fila de ACK.

### Sprint B (P1): politica remota de execucao

Escopo:

1. criar consumo de policy remota para intervalo/jitter/backoff com fallback local.
2. registrar policy aplicada e tempo por fase em `agentMetrics`.
3. revisar thresholds de coleta (software/system/network) com base em policy.

Definicao de pronto:

- intervalo do ciclo pode ser ajustado no portal sem alterar pacote local.
- metricas de ciclo mostram policy aplicada no host.
- fallback local preserva operacao offline de configuracao.

### Sprint C (P1): hardening e observabilidade

Escopo:

1. padronizar `correlationId` por ciclo em todas as chamadas HTTP do agente.
2. aumentar sinais operacionais no log para erros de autenticacao/rate limit/timeout.
3. alinhar codigos de warning com o dominio remoto para triagem mais rapida.

Definicao de pronto:

- chamadas `discover/bootstrap/sync/ack` podem ser correlacionadas fim a fim.
- suporte consegue diagnosticar falha de ciclo sem acesso interativo ao host.
- taxa de falso positivo em rebootstrap reduzida.

### Sprint D (P1): inventario tecnico Syspro/Firebird consolidado

Escopo:

1. consolidar no ciclo de `sync` os campos de instalacao por base:
- `isServerHost`
- `hasClientFolder`
- `hasDllFolder`
- `firebirdVersion`
- `firebirdPath`
2. garantir persistencia unificada desses campos no backend e consumo no portal.
3. eliminar fallback desnecessario quando dado persistido estiver disponivel.

Definicao de pronto:

- aba `Informacoes tecnicas` mostra leitura coerente para servidor Syspro e Firebird.
- aba `Instalacoes` usa metadados persistidos por instalacao para diagnostico operacional.
- contrato agente/API documentado com campos opcionais e sem quebra retrocompativel.
