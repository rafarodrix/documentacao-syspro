## Objetivo

Consolidar a operacao do agente remoto RustDesk e definir a migracao do script monolitico (`trilink-agente.ps1`) para arquitetura modular, sem quebrar o fluxo atual `discover -> bootstrap -> sync -> ack`.

## Estado atual (2026-04-02)

- A modularizacao do agente (`FEAT-002J`) foi concluida e o `trilink-agente.ps1` agora atua como orquestrador de ciclo.
- O contrato de `sync` no dominio remoto ja aceita inventario tecnico expandido.
- A persistencia em `remote_host` ja foi ampliada para os novos blocos de inventario.
- O diretorio e o detalhe de host no portal ja exibem sinais operacionais derivados do inventario.

## Plano de implementacao recomendado (proximas sprints)

### Sprint A (P0): estabilidade operacional do ciclo

Escopo:

1. ajustar orquestracao no `trilink-agente.ps1` para priorizar `sync` com token valido e reduzir bootstrap recorrente.
2. manter invalidacao de token apenas apos finalizar a fila de ACK (sem quebrar comandos subsequentes).

Status atual:

- `UPGRADE_CLIENT` implementado em `application/commands.ps1` com download HTTPS, validacao SHA256 e ACK detalhado.
- portal atualizado para so emitir `upgrade_client` quando metadados seguros de upgrade estiverem presentes.

Definicao de pronto:

- comando `UPGRADE_CLIENT` tem caminho de sucesso e falha cobertos.
- ciclo nominal de host ativo conclui sem bootstrap desnecessario.
- testes locais do fluxo de comandos passam.

### Sprint B (P1): politica remota de execucao

Escopo:

1. criar consumo de policy remota para intervalo/jitter/backoff com fallback local.
2. registrar policy aplicada e tempo por fase em `agentMetrics`.
3. revisar thresholds de coleta (software/system/network) com base em policy.

Definicao de pronto:

- intervalo do ciclo pode ser ajustado no portal sem alterar pacote local.
- metricas de ciclo mostram policy aplicada no host.
- fallback local preserva operacao offline de configuracao.

### Sprint C (P1): hardening e observabilidade

Escopo:

1. padronizar `correlationId` por ciclo em todas as chamadas HTTP do agente.
2. aumentar sinais operacionais no log para erros de autenticacao/rate limit/timeout.
3. alinhar codigos de warning com o dominio remoto para triagem mais rapida.

Definicao de pronto:

- chamadas `discover/bootstrap/sync/ack` podem ser correlacionadas fim a fim.
- suporte consegue diagnosticar falha de ciclo sem acesso interativo ao host.
- taxa de falso positivo em rebootstrap reduzida.
