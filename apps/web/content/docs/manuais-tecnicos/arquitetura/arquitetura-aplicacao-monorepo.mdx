## Objetivo

Definir a arquitetura oficial da aplicacao para garantir:

- escalabilidade tecnica
- separacao previsivel de responsabilidades
- evolucao segura para mobile e backend separado
- reducao de acoplamento entre UI, regras e infraestrutura

## Diagnostico da situacao atual

Hoje o projeto ainda e, na pratica, um **modular monolith em Next.js**, com boa parte da regra ja concentrada em `src/core`, mas com estes gaps:

- mistura de responsabilidades entre `app`, `actions`, `components` e `core`
- features ainda parcialmente espalhadas fora de `src/features`
- contratos ainda nao centralizados em um pacote proprio
- acesso a infraestrutura ainda acoplado ao app web em varios fluxos
- monorepo alvo documentado, mas ainda sem fase intermediaria formal

## Avaliacao da arquitetura desejada

### O que esta certo

A arquitetura alvo com:

- `apps/web`
- `packages/api`
- `packages/core`
- `packages/contracts`
- `packages/database`
- `packages/shared`
- `packages/ui`

e **boa e escalavel** para:

- multi-app (`web`, `mobile`, `workers`)
- backend separado no futuro
- compartilhamento de contratos
- evolucao por times e dominios

### O que precisa melhorar

O ponto a ajustar nao e a arquitetura final, e a **sequencia de migracao**.

Ir direto para:

- workspaces
- extracao de todos os packages
- criacao de `apps/web`
- API dedicada

em um unico movimento **nao e a melhor estrategia** para este projeto.

Para escalar com menos risco, a arquitetura correta precisa ter **4 estagios**:

1. **Fase A - Modular monolith disciplinado**
   - consolidar `src/features/<feature>/domain|application|interface`
   - remover duplicacao de regra entre dashboard, actions, pages e components
   - estabelecer contratos internos e boundaries

2. **Fase B - Monorepo estrutural**
   - criar `apps/web`
   - criar `packages/shared`, `packages/ui`, `packages/contracts`
   - mover apenas o que ja estiver estabilizado

3. **Fase C - Monorepo de dominio**
   - extrair `packages/core`
   - extrair `packages/database`
   - reduzir dependencia direta de `Next.js` e `Server Actions`

4. **Fase D - Multi-app**
   - adicionar `apps/mobile`
   - adicionar `apps/api` ou `apps/bff`
   - mover casos de uso compartilhados para consumo externo

## Arquitetura recomendada para escala

### Fase A - Arquitetura recomendada agora

Antes de extrair pacotes, a aplicacao deve ficar assim:

- `src/features/<feature>/domain`
  - tipos, enums, regras puras, mapeamentos sem efeito colateral
- `src/features/<feature>/application`
  - queries, actions, orchestration, facades e mapeamento para view model
- `src/features/<feature>/interface`
  - componentes React, containers, tabelas, modais

Camadas complementares:

- `src/core/domain`
  - entidades e contratos realmente transversais
- `src/core/infrastructure`
  - gateways, cache, observabilidade, adapters
- `src/actions`
  - apenas ponte de transporte para chamar a camada de application
- `src/app`
  - roteamento e composicao de tela

### Fase B - Arquitetura alvo de monorepo

- `apps/web`
  - Next.js App Router
  - rotas, layouts, providers e composicao de interface
- `packages/contracts`
  - DTOs, schemas Zod e payloads de fronteira
- `packages/shared`
  - utilitarios, helpers, constants e funcoes puras compartilhadas
- `packages/ui`
  - design system e componentes reutilizaveis
- `packages/core`
  - casos de uso, dominio, politicas, regras de negocio
- `packages/database`
  - Prisma schema, migrations, client e repositories/adapters
- `packages/api`
  - tRPC ou BFF, authz, context, middlewares e roteadores

### Fase C - Arquitetura para mobile e servidor separado

Quando houver `apps/mobile` ou `apps/api`, o consumo ideal fica:

- `apps/web` -> `packages/contracts`, `packages/ui`, `packages/core`
- `apps/mobile` -> `packages/contracts`, `packages/core`, `packages/shared`
- `apps/api` -> `packages/contracts`, `packages/core`, `packages/database`

## Regras de dependencia

### Dentro do app atual

- fluxo permitido: `interface -> application -> domain`
- `interface` nao importa `prisma`, `@/lib/prisma` ou gateway tecnico diretamente
- `application` pode chamar infraestrutura via adapter/gateway
- `domain` nao depende de React, Next.js, Prisma ou transporte

### No monorepo alvo

- `apps/*` nao concentram regra de negocio
- `packages/contracts` nao depende de framework
- `packages/core` nao depende de Next.js
- `packages/database` nao depende de UI
- `packages/ui` nao depende de Prisma nem auth

## Padrao oficial de data fetching e invalidacao

O padrao oficial do projeto passa a ser:

### Leitura inicial de tela

- `page.tsx` e containers server-side devem buscar dados via `src/features/<feature>/application/queries.ts`
- a query monta o view model da tela e pode usar Prisma direto apenas quando a leitura ainda for local e simples
- a tela nao deve duplicar filtro, agregacao ou mapeamento ja existente em `application`

### Mutacao

- componente client chama `src/features/<feature>/application/actions.ts`
- `actions` concentram escrita, authz e invalidacao
- `page.tsx` e componente React nao executam `revalidatePath` nem `revalidateTag`

### Widgets dinamicos, polling e observabilidade

- `fetch('/api/...')` permanece valido para:
  - polling
  - barra de status
  - observabilidade
  - integracoes que exigem refresh em tempo real
- nesses casos, o padrao recomendado e `cache: 'no-store'` ou controle explicito de `revalidate`

### Invalidacao

- invalidacao de cache/path deve ser centralizada em helper unico
- o helper oficial atual fica em `src/lib/cache-invalidation.ts`
- cada dominio invalida apenas seu conjunto minimo de telas:
  - `revalidateCadastrosViews`
  - `revalidateContractsViews`
  - `revalidateSettingsViews`
  - `revalidateTaxViews`
  - `revalidateTicketCollections`
  - `revalidateTicketViews`
  - `revalidateDocumentosViews`

### Anti-padroes de fetching

1. `page.tsx` chamar action de escrita para ler dados
2. componente client duplicar query existente em `application/queries`
3. cada action declarar sua propria lista de `revalidatePath`
4. usar `fetch('/api/...')` para leitura inicial de pagina quando a query ja pode rodar no server
5. misturar invalidacao de contratos, cadastros e settings sem helper central

## Estrategia de repositories e adapters para persistencia

O objetivo nao e criar repositories por formalismo. O objetivo e impedir que o dominio e os casos de uso fiquem acoplados a Prisma quando a aplicacao evoluir para `packages/core`, `packages/database` e consumo por API/mobile.

### Regra principal

Enquanto estivermos no **modular monolith**, a regra recomendada e:

- `application` pode usar um **adapter Prisma local** quando o caso for simples
- `application` deve passar a usar um **repository contract** quando a regra de negocio:
  - agrega mais de uma tabela
  - possui filtro/tenant/authz reutilizavel
  - precisa de teste mais isolado
  - sera reaproveitada por web, API, worker ou mobile

Ou seja:

- nem tudo vira repository agora
- mas toda feature critica deve nascer pronta para migrar para repository sem retrabalho estrutural

### Padrao recomendado por camada

- `domain`
  - define tipos de entidade, regras puras e contratos de persistencia quando necessario
- `application`
  - orquestra casos de uso
  - depende de interfaces de persistencia para fluxos criticos
- `infrastructure`
  - implementa adapters concretos com Prisma, cache, gateway externo ou storage

### Contrato recomendado

Para cada feature critica, o contrato deve ficar proximo do dominio da feature, por exemplo:

- `src/features/tickets/domain/repositories/ticket-read.repository.ts`
- `src/features/company/domain/repositories/company-read.repository.ts`
- `src/features/company/domain/repositories/company-write.repository.ts`

As implementacoes concretas ficam em infraestrutura, por exemplo:

- `src/core/infrastructure/persistence/prisma/tickets/prisma-ticket-read.repository.ts`
- `src/core/infrastructure/persistence/prisma/company/prisma-company-read.repository.ts`
- `src/core/infrastructure/persistence/prisma/company/prisma-company-write.repository.ts`

### Quando usar query direta com Prisma

Pode continuar usando Prisma direto em `application` quando:

- a leitura e simples e local
- nao existe reutilizacao relevante
- o fluxo ainda esta mudando rapido
- o custo de abstracao e maior que o beneficio

Exemplos tipicos:

- combo de opcoes para select
- leitura administrativa simples
- consulta temporaria de tela sem reuso

### Quando extrair repository obrigatoriamente

Deve virar repository quando houver:

1. regra de escopo por tenant/empresa/role reutilizada
2. agregacao entre varias tabelas
3. composicao com cache
4. fallback entre banco e provider externo
5. uso pelo dashboard e pela listagem ao mesmo tempo
6. necessidade de mover o caso de uso para `packages/core`

### Estrategia concreta por modulo

#### Tickets

`tickets` deve ser o primeiro modulo a formalizar repositories, porque ja possui:

- query service compartilhado
- cache
- gateway externo
- dashboard e listagem consumindo o mesmo dominio

Padrao sugerido:

- `TicketReadRepository`
- `TicketCacheRepository`
- `TicketGatewayAdapter`

Status atual do recorte:

- contratos criados em `src/features/tickets/domain/repositories`
- `IZammadGateway` segue como adapter tecnico em `core/infrastructure`
- `ticket-query-service` continua como orquestrador, agora com contrato-alvo explicito para a proxima extracao

#### Company

`company` pode continuar com Prisma em `queries` simples, mas deve extrair repository para:

- carregamento da empresa com memberships e escopo
- e-mails Zammad da empresa
- operacoes que misturam empresa, endereco e vinculos

Status atual do recorte:

- contratos criados em `src/features/company/domain/repositories`
- leituras simples continuam podendo usar Prisma direto em `application/queries`
- escritas e lookup externo ficam candidatas a adapter/repository concreto quando forem movidos para `packages/core`

#### Tax

`tax` deve separar:

- repositorio de leitura local
- adapter de sincronizacao externa
- writer de persistencia por lote

Porque o modulo mistura:

- consulta local
- sync externo
- regras de normalizacao

Status atual do recorte:

- contratos criados em `src/features/tax/domain/repositories`
- `queries` locais continuam aceitas com Prisma direto
- batches de sync ficam mapeados como responsabilidade de `TaxWriteRepository`

### Convencao oficial de pastas

Contratos por feature:

- `src/features/<feature>/domain/repositories/*.repository.ts`

Implementacoes concretas enquanto o app ainda estiver em `apps/web`:

- `src/core/infrastructure/persistence/prisma/<feature>/*.repository.ts`
- `src/core/infrastructure/cache/<feature>/*.repository.ts`
- `src/core/infrastructure/gateways/*` para providers externos

Quando um fluxo ainda nao justificar repository, o uso direto de Prisma fica restrito a:

- `src/features/<feature>/application/queries.ts`
- `src/features/<feature>/application/actions.ts`

e nunca em:

- `interface`
- `app`
- `components`
- `packages/core`
- carga incremental

#### Settings e contracts

`settings/contracts` podem ficar sem repository formal no curto prazo, desde que:

- `queries` e `actions` permaneçam finas
- tipos fiquem centralizados
- nao voltem a acessar infraestrutura fora da feature

### Regra de migracao para Fase C

Ao entrar em `INFRA-016` e `INFRA-017`, o objetivo nao sera reescrever tudo.

O movimento correto sera:

1. promover contratos de persistencia criticos para `packages/core`
2. mover implementacoes Prisma para `packages/database`
3. manter queries simples no app web enquanto nao justificarem extracao

### Anti-padroes de persistencia

1. criar repository generico CRUD sem semantica de dominio
2. esconder Prisma atras de interface inutil
3. colocar authz, tenant e regra de negocio dentro do componente
4. deixar `page.tsx` montar query complexa de banco
5. misturar gateway externo e persistencia local no mesmo arquivo sem contrato claro

## O que e melhor nao extrair agora

Para escalar com menos risco, **nao vale extrair imediatamente**:

- `packages/api`
- `packages/database`
- `apps/mobile`
- `apps/api`

Esses itens devem vir **depois** que:

- features criticas estiverem em `src/features`
- contratos tiverem estabilizado
- a aplicacao parar de depender de varios atalhos internos

## Prioridade arquitetural recomendada

### Prioridade 1

- consolidar features no padrao `domain/application/interface`
- transformar `actions` em ponte fina
- criar contratos/view models compartilhados por feature
- eliminar logica duplicada entre dashboard, page e components

### Prioridade 2

- criar workspaces
- mover `web` para `apps/web`
- extrair `packages/shared`, `packages/ui`, `packages/contracts`

### Prioridade 3

- extrair `packages/core`
- extrair `packages/database`
- introduzir camada `packages/api`

### Prioridade 4

- mobile
- backend dedicado
- workers/filas

## Guardrails de implementacao

1. Nao criar regra de negocio em componente React.
2. Nao duplicar filtros, mapeamentos e KPIs entre telas.
3. Toda feature nova deve nascer em `src/features/<feature>`.
4. `actions` devem delegar para `application`, nao carregar regra propria.
5. Nao mover para `packages/*` o que ainda muda toda semana.
6. Extracao para monorepo deve ser por bloco estavel, nao por entusiasmo arquitetural.

## Checklist de revisao arquitetural

- Feature criada em `src/features/<feature>` com 3 camadas
- Pagina consome `application`, nao regra espalhada
- `actions` finas e sem duplicacao
- contratos e view models centralizados por feature
- gateways tecnicos isolados em infraestrutura
- nenhuma tela recalcula regra ja existente em outro ponto

## Anti-padroes

1. Colocar regra em componente React.
2. Colocar logica de negocio dentro de `page.tsx`.
3. Duplicar query de ticket, KPI ou filtro em mais de uma tela.
4. Extrair pacote cedo demais e carregar mais custo operacional que beneficio.
5. Misturar auth, tenant e regra de dominio no mesmo componente.
6. Criar API dedicada antes de estabilizar contratos internos.

## Conclusao

A arquitetura documentada **e boa como arquitetura alvo**, mas a melhor versao para escalar este projeto e:

- **agora:** modular monolith disciplinado por feature
- **depois:** monorepo estrutural
- **por fim:** multi-app e backend separado

Ou seja: a arquitetura final continua valida, mas o plano correto e **evolucao em etapas**, nao migracao total imediata.
