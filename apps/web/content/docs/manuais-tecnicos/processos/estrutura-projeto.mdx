## Objetivo

Descrever a estrutura que de fato esta em uso hoje no projeto, evitando referencia a layouts antigos que ja nao representam a base atual.

## Visao geral do monorepo

```text
apps/
  web/        -> portal, docs, area publica e operacao principal
  api/        -> shell HTTP dedicado para evolucao do backend
  mobile/     -> shell estrutural para runtime mobile futuro
packages/
  api/        -> nucleo modular do BFF
  contracts/  -> DTOs, schemas e contratos de fronteira
  core/       -> regras e casos de uso extraiveis do app
  database/   -> schema Prisma, client e persistencia compartilhada
  shared/     -> utilitarios e funcoes puras compartilhadas
  ui/         -> componentes e design system reutilizavel
```

## Estrutura principal de `apps/web`

```text
apps/web/
  src/
    app/          -> roteamento, layouts e endpoints do App Router
    features/     -> dominio funcional por modulo
    components/   -> componentes compartilhados de UI e composicao
    lib/          -> infraestrutura transversal do app web
    providers/    -> providers React
    types/        -> tipos locais de suporte
  content/
    docs/         -> conteudo MDX da documentacao
  public/         -> assets estaticos
```

## Regra por camada no app web

### `src/app`

Responsavel por:

- rotas
- layouts
- composicao de tela
- route handlers do App Router

Nao deve concentrar:

- regra de negocio
- consulta complexa repetida
- integracao tecnica espalhada

### `src/features/<feature>`

Cada feature deve evoluir com separacao previsivel:

```text
src/features/<feature>/
  domain/          -> tipos, regras puras, contratos
  application/     -> queries, actions, orchestration, view models
  infrastructure/  -> gateways, cache, adapters e mapeamentos tecnicos
  interface/       -> componentes, tabelas, modais e containers
  hooks/           -> hooks client-side especificos da feature
```

### `src/components`

Fica reservado para:

- `ui/` com componentes reutilizaveis de base
- composicoes transversais que nao pertencem a uma unica feature

Nao deve virar deposito de regra de negocio.

### `src/lib`

Camada transversal do app web para itens como:

- auth
- prisma client local
- invalidacao de cache
- observabilidade
- helpers tecnicos compartilhados

Regra funcional de dominio nao deve nascer aqui se pertencer claramente a uma feature.

## Estrutura de docs

```text
apps/web/content/docs/
  docs/                 -> documentacao publica e funcional
  manuais-tecnicos/     -> arquitetura, governanca, processos e estrategia
```

Dentro de `manuais-tecnicos`, a organizacao correta e:

- `arquitetura/` para fonte de verdade estrutural
- `governanca/` para checklist, backlog e guardrails
- `processos/` para manuais operacionais
- `estrategia/` para direcao futura de modulos relevantes

## Onde colocar codigo novo

### Regra funcional nova

Criar ou expandir uma feature em `apps/web/src/features/<feature>`.

### Contrato compartilhado entre apps

Avaliar `packages/contracts`.

### Regra de negocio com potencial de extracao

Avaliar `packages/core`, desde que o recorte ja esteja estavel.

### Persistencia compartilhada

Avaliar `packages/database`.

### Componente visual reutilizavel

Avaliar `packages/ui` ou `apps/web/src/components/ui`, conforme o escopo de reuso.

## Sinais de estrutura antiga que nao devem voltar

- pasta `server/` como centro de regra de negocio do app
- components organizados por area antiga em vez de por feature
- paginas montando query e autorizacao manualmente
- helpers de dominio espalhados em `lib/`
- documentos descrevendo apenas `app/`, `components/` e `lib/` como se o monorepo nao existisse
