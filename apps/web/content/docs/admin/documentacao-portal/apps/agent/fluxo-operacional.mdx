# Agent - Fluxo Operacional

> Como o agente se registra, opera e executa comandos do portal. Atualizado em: 2026-07-13

---

## Ciclo de vida completo

```text
Instalacao
  -> Discovery
  -> Vinculo no portal
  -> Bootstrap autenticado
  -> Sync continuo
  -> ACK de comandos
```

### Resumo do fluxo

1. O agente faz `discover` com `discoveryToken`.
2. O portal registra a maquina descoberta e informa o estado operacional.
3. Depois do vinculo do host, o `discover` passa a devolver `installToken`.
4. O agente usa esse `installToken` para executar `bootstrap`.
5. O portal devolve `agentToken` e a configuracao esperada do RustDesk.
6. O agente persiste o `agentToken` localmente com DPAPI.
7. O reconcile passa a executar `sync` continuo com snapshots e metricas.

> Nota sobre bootstrap: o agente nao usa mais `REMOTE_INSTALL_TOKEN` local.
> O `bootstrap` so acontece quando o `discover` devolve `installToken`
> do portal para um host ja vinculado.

> Nota sobre heartbeat vs sync: `AgentDevice.lastHeartbeatAt` e atualizado
> por `POST /api/agents/heartbeat`. Ja `RemoteHost.lastHeartbeatAt`,
> metricas e snapshots operacionais sao atualizados por
> `POST /api/remote/rustdesk/sync`.

## Fase 1 - Discovery

O agente usa `discoveryToken` para se apresentar pela primeira vez.
Esse fluxo serve para triagem inicial e para descobrir quando um host
ja esta vinculado e apto a bootstrap.

### Payload enviado

```json
{
  "schemaVersion": "discover.payload.v1",
  "discoveryToken": "disc_abc123",
  "rustdeskId": "123456789",
  "machineName": "PC-CLIENTE-01",
  "agentVersion": "go-agent-v1"
}
```

### Respostas operacionais

| `bootstrapFlow` | Significado | Proximo passo |
| --- | --- | --- |
| `pending_link` | Maquina descoberta, mas ainda sem host vinculado | Aguardar vinculo |
| `linked_host_detected` | O `RemoteDiscoveredHost` ja esta vinculado no portal e segue aguardando ou reaproveitando bootstrap | Aguardar `installToken` |
| `host_bootstrap_required` | O `RemoteDiscoveredHost` ja vinculado requer bootstrap autenticado | Bootstrap |
| `token_invalid` | O `RemoteDiscoveredHost` ja vinculado tem credencial remota invalida ou rotacionada | Bootstrap |

Se o host/dispositivo tiver sido removido do portal, o backend passa a bloquear a rematerializacao automatica no `discover`. Nesse caso o agente continua sem reaparecer ate nova autorizacao operacional ou reinstalacao controlada.

O backend tambem nao faz mais autolink implicito por `rustdeskId` para hosts
legados. Mesmo que exista um `RemoteHost` antigo com o mesmo identificador, a
nova instalacao entra em `pending_link` ate existir `RemoteDiscoveredHost`
explicito e vinculo/manual autorizacao no portal.

## Fase 2 - Bootstrap

O agente troca o `installToken` recebido do portal por um `agentToken`
permanente.

Se o `discover` nao devolver `installToken`, o agente espera o proximo
ciclo e nao tenta reaproveitar token local legado.

O backend agora so aceita `bootstrap` quando esse `installToken` estiver
amparado por um `discover` recente da mesma maquina vinculada. Token de
instalacao sem triagem recente, ou vindo de identidade divergente, e rejeitado.

Cada `discover` que exigir bootstrap emite um novo `installToken`, e o token
anterior deixa de valer. Depois do bootstrap bem-sucedido, esse `installToken`
tambem e invalidado no portal.

### Payload enviado

```json
{
  "installToken": "rhost_...",
  "rustdeskId": "123456789",
  "machineName": "PC-CLIENTE-01",
  "agentVersion": "go-agent-v1",
  "currentAlias": "PC-CLIENTE-01"
}
```

### Resposta esperada

```json
{
  "agentToken": "agt_...",
  "agentTokenExpiresAt": "2027-05-05T00:00:00Z",
  "serverHost": "relay.trilink.com.br",
  "apiHost": "api.trilink.com.br",
  "alias": "PC-CLIENTE-01"
}
```

O `agentToken` e persistido localmente com DPAPI e passa a ser usado
nas chamadas de `sync` e `ack`.

## Fase 3 - Heartbeat separado

O heartbeat de `POST /api/agents/heartbeat` mantem o cadastro do
`AgentDevice` atualizado, mas nao publica snapshots do host remoto.

### Payload enviado

```json
{
  "deviceId": "WIN-ABC123",
  "agentVersion": "go-agent-v1",
  "remoteLinkContext": {
    "remoteHostId": "host_...",
    "companyId": "company_...",
    "rustdeskId": "123456789"
  }
}
```

### Efeito persistido

- Atualiza `AgentDevice.lastHeartbeatAt`
- Mantem vinculo entre `AgentDevice` e `RemoteHost`
- Nao cria nem reativa `RemoteDiscoveredHost` por fora do ingress oficial
- Nao substitui o `sync` remoto

Se uma maquina removida do portal precisar voltar, a retomada deve passar pelo
fluxo oficial de `discover`/`bootstrap` apos reinstalacao ou reautorizacao
operacional. `register`/`heartbeat` isolados nao rematerializam mais o host.

## Fase 4 - Reconcile e Sync

O reconcile executa em ciclos e roda os modulos em ordem:

1. `device`
2. `remote`
3. `tunnel`
4. `backup`
5. `support`

### Device module

Coleta snapshots como:

- `agentMetrics`
- `diskSnapshot`
- `sysproProcesses`
- `sysproVersions`
- `windowsUpdateStatus`
- `hardwareIdentity`

### Remote module

Envia `POST /api/remote/rustdesk/sync` com:

- `agentToken`
- `rustdeskId`
- `agentMetrics`
- snapshots de disco, processos e versoes
- `sysproUpdates`

### Efeito persistido no portal

- Atualiza `RemoteHost.lastHeartbeatAt`
- Atualiza `RemoteHost.lastHeartbeatSuccessAt`
- Atualiza `RemoteHost.lastAgentMetrics`
- Atualiza snapshots operacionais do host
- Reentrega comandos pendentes e registra comandos entregues

---

## Fase 5 - ACK de comandos

Depois de executar comandos devolvidos pelo `sync`, o agente envia:

```json
{
  "schemaVersion": "ack.payload.v1",
  "agentToken": "agt_...",
  "commandId": "cmd_001",
  "status": "ACKNOWLEDGED",
  "reasonCode": "COMMAND_PROCESSED",
  "message": "Configuracao reaplicada com sucesso"
}
```

---

## Por que um host pode aparecer sem metricas

1. O agente nunca completou o bootstrap e, portanto, nao possui `agentToken`.
2. O `sync` nao esta executando com sucesso.
3. O portal ainda nao devolveu `installToken` depois do vinculo.
4. O host esta offline ha muito tempo e nunca chegou a publicar snapshots validos.

## Leitura operacional no portal

| Sinal | Origem real |
| --- | --- |
| Ultimo contato do dispositivo | `AgentDevice.lastHeartbeatAt` |
| Ultimo contato da maquina remota | `RemoteHost.lastHeartbeatAt` |
| CPU e RAM do host remoto | `RemoteHost.lastAgentMetrics` |
| Status de convergencia RustDesk | resposta e compliance do `sync` |

## Comandos remotos suportados

| Tipo | Efeito |
| --- | --- |
| `REAPPLY_ALIAS` | Reaplica alias do RustDesk |
| `REAPPLY_CONFIG` | Reaplica configuracao esperada do RustDesk |
| `UPGRADE_CLIENT` | Baixa e instala nova versao do RustDesk |
| `ROTATE_TOKEN_REQUIRED` | Solicita novo bootstrap/token no proximo ciclo |
