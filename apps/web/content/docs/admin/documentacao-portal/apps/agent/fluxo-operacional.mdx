O discovery continua existindo, mas agora deve ser lido como parte da capability remota, nĂŁo da identidade da instalaĂ§ĂŁo.

### Respostas operacionais

| `bootstrapFlow` | Significado | Proximo passo |
| --- | --- | --- |
| `pending_link` | Maquina descoberta e apta ao bootstrap tecnico local, mas ainda sem host/empresa vinculados | Executar bootstrap tecnico e aguardar vinculo |
| `linked_host_detected` | O `RemoteDiscoveredHost` ja esta vinculado no portal e segue aguardando ou reaproveitando bootstrap | Aguardar `installToken` |
| `host_bootstrap_required` | O `RemoteDiscoveredHost` ja vinculado requer bootstrap autenticado | Bootstrap |
| `token_invalid` | O `RemoteDiscoveredHost` ja vinculado tem credencial remota invalida ou rotacionada | Bootstrap |

## Fase 4 - Bootstrap da capability remota

O `installToken` e o `agentToken` passam a ser credenciais da capability remota.

Quando o `discover` responder `pending_link`, o agente ja pode chamar o
bootstrap tecnico usando `discoveryToken + discoveredHostId` para instalar ou
reconfigurar o RustDesk antes do vinculo empresarial.

Se o `discover` nao devolver `installToken` em fluxo vinculado, o agente espera
o proximo ciclo e nao tenta reaproveitar token local legado.

O backend agora so aceita `bootstrap` quando esse `installToken` estiver
amparado por um `discover` recente da mesma maquina vinculada. Token de
instalacao sem triagem recente, ou vindo de identidade divergente, e rejeitado.

No fluxo tecnico (`pending_link`), o backend exige `discover` recente da mesma
maquina e `discoveryToken` valido, mas ainda nao emite `agentToken`. O sync
autenticado continua reservado para depois do vinculo.

Cada `discover` que exigir bootstrap emite um novo `installToken`, e o token
anterior deixa de valer. Depois do bootstrap bem-sucedido, esse `installToken`
tambem e invalidado no portal.

### Payload enviado

```json
{
  "discoveryToken": "DISCOVERY_TOKEN",
  "discoveredHostId": "disc_...",
  "rustdeskId": "123456789",
  "machineName": "PC-CLIENTE-01",
  "agentVersion": "go-agent-v1",
  "currentAlias": "PC-CLIENTE-01"
}
```

### Resposta esperada

```json
{
  "agentToken": "agt_...",
  "agentTokenExpiresAt": "2027-05-05T00:00:00Z",
  "serverHost": "relay.trilink.com.br",
  "apiHost": "api.trilink.com.br",
  "alias": "PC-CLIENTE-01"
}
```

O `agentToken` continua protegido localmente com DPAPI, mas nĂŁo representa a instalaĂ§ĂŁo inteira do agent. Ele representa sĂł a comunicaĂ§ĂŁo remota desse host/capability com o portal.

O estado remoto persistido pelo agent tambem protege com DPAPI as senhas
locais do RustDesk usadas pelo runtime (`default_password` e
`runtime_password`), reduzindo exposicao de credenciais no disco.

O runtime agora valida explicitamente os contratos remotos devolvidos pelo
portal (`discover.v2`, `rustdesk.bootstrap.v1` e `rustdesk.sync.v1`). Resposta
incompleta, versao inesperada ou `installToken` fora de fluxo esperado passam
a falhar fechado e entram no estado estruturado de erro local.

## Fase 5 - Reconcile e sync das capabilities

O reconcile executa em ciclos e roda os modulos em ordem:

1. `device`
2. `remote`
3. `tunnel`
4. `backup`
5. `support`

### Device module

Coleta snapshots como:

- `agentMetrics`
- `diskSnapshot`
- `sysproProcesses`
- `sysproVersions`
- `windowsUpdateStatus`
- `hardwareIdentity`

### Remote module

Enquanto a capability remota ainda mora em `internal/modules/remote`, ela envia `POST /api/remote/rustdesk/sync` com:

- `agentToken`
- `rustdeskId`
- `agentMetrics`
- snapshots de disco, processos e versoes
- `sysproUpdates`

### Efeito persistido no portal

- Atualiza `RemoteHost.lastHeartbeatAt`
- Atualiza `RemoteHost.lastHeartbeatSuccessAt`
- Atualiza `RemoteHost.lastAgentMetrics`
- Atualiza snapshots operacionais do host
- Reentrega comandos pendentes e registra comandos entregues

No runtime local, os eventos criticos de operacao agora entram em uma
`telemetry_outbox.json` persistida. O agent deixa de depender apenas de buffer
em memoria para manter a trilha de `discover`, `bootstrap`, `sync`,
`heartbeat` e `reconcile`.

## Fase 6 - ACK de comandos

Depois de executar comandos devolvidos pelo `sync`, o agente envia:

```json
{
  "schemaVersion": "ack.payload.v1",
  "agentToken": "agt_...",
  "commandId": "cmd_001",
  "status": "ACKNOWLEDGED",
  "reasonCode": "COMMAND_PROCESSED",
  "message": "Configuracao reaplicada com sucesso"
}
```

Se o ACK falhar, o agent grava a confirmacao em `pending_ack_queue.json` com
token protegido, backoff persistido, proxima tentativa e ultimo erro
estruturado. O replay deixa de depender apenas do processo atual.

1. A instalaĂ§ĂŁo existe, mas a capability remota nunca completou bootstrap e, portanto, nĂŁo possui `agentToken`.
2. O `sync` nao esta executando com sucesso.
3. O portal ainda nao devolveu `installToken` depois do vinculo.
4. O host esta offline ha muito tempo e nunca chegou a publicar snapshots validos.
