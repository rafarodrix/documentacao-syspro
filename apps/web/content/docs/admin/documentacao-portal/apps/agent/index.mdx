## Responsabilidade

O `apps/agent` é uma aplicação desktop para Windows que:
- Resolve a identidade física do dispositivo (`device_id`)
- Mantém uma identidade de instalação local independente (`agent_instance_id`)
- Orquestra capabilities gerenciadas pelo core do agente
- Expõe uma UI local (Wails/React) para onboarding, suporte e estado operacional
- Publica heartbeat base da instalação e sync operacional das capabilities

---

## Dois binários independentes

O agente compila em **dois binários distintos** que se comunicam via IPC:

```
agent-service.exe
  ├── Roda como Windows Service (LocalSystem)
  ├── Executa todo o trabalho crítico em background
  ├── Servidor IPC (named pipe)
  └── Responsabilidades:
      - identidade do dispositivo e da instalação
      - enrollment, heartbeat e desired state
      - reconcile de capabilities e jobs tipados
      - módulos atuais: remote, backup, device, support, tunnel

agent-ui.exe  (alias: agent.exe)
  ├── Aplicação desktop Wails (WebView2 + React)
  ├── Exibe estado, chat e pipeline de provisionamento
  ├── Cliente IPC (conecta ao agent-service)
  └── Responsabilidades:
      - UI de onboarding e status
      - Timeline de progresso
      - Chat de suporte ao vivo
```

## Modelo de identidade

O novo escopo do agente separa três conceitos:

```text
Device
  └── representa a maquina fisica
      - device_id
      - hostname
      - os
      - identity_source

Agent Installation
  └── representa uma instalacao local especifica do agent
      - agent_instance_id
      - credential_id
      - installed_at

Capability
  └── representa um componente externo gerenciado pelo core
      - remote access / RustDesk
      - backup / Firebird
      - tunnel / Rathole
      - upload / Rclone
```

### Estado atual do corte

- O runtime local já persiste `identity.json` para o `device_id`.
- O runtime local agora também persiste `installation.json` com `agent_instance_id` e `credential_id`.
- O backend agora persiste a presença base em `Device` e `AgentInstallation`.
- A capability remota agora passa a ser materializada em `AgentCapability`, ligada ao `RemoteHost`.
- O banco passa a tratar a presença base do agent por `Device` e `AgentInstallation`, sem manter tabela paralela de legado.

## Capabilities e módulos

### Core do agente

- O core do agente já assume `device_id` e `agent_instance_id` como identidades diferentes.
- `POST /api/agents/register` e `POST /api/agents/heartbeat` deixam de representar a capability remota; eles representam a presença da instalação.
- O estado desejado continua sendo reconciliado em ciclos, mas a meta do refactor é convergir para um `Component Manager`.

### Capability remota / RustDesk

**Path atual:** `internal/modules/remote/`

A capability remota ainda está concentrada no módulo `remote`, mas o novo escopo passa a tratá-la como um componente externo gerenciado pelo core, e não como identidade principal do agente.

Hoje ela ainda acumula:

- discover / bootstrap / sync / ack
- token remoto do host
- aplicação e leitura de configuração do RustDesk
- instalação/upgrade do cliente remoto
- fila de comandos remotos

Esse acoplamento é exatamente o alvo da próxima extração para uma fronteira explícita de capability.

### Outras capabilities atuais

- `backup`: backup Firebird via `gbak`
- `device`: snapshots de hardware, sistema, discos, serviços e versões
- `support`: contexto de suporte e Chatwoot local
- `tunnel`: espaço reservado para túnel gerenciado

## IPC (comunicação service ↔ ui)

**Path:** `internal/infra/ipc/`

| Componente       | Protocolo                             |
|------------------|---------------------------------------|
| Windows          | Named Pipe (`\\.\pipe\trilink-agent`) |
| Outros (dev)     | Unix Domain Socket                    |

**Protocolo:** mensagens JSON sobre pipe. Cada mensagem tem tipo + payload.

---

## Secrets e segurança

- Segredos persistidos localmente continuam protegidos com **DPAPI** no Windows.
- `credential_id` identifica a credencial ativa da instalação, mas não substitui o armazenamento protegido dos segredos reais.
- `agentToken` remoto continua sendo tratado como credencial da capability remota, não da identidade base do agente.
- Em não-Windows (dev): arquivo simples em `/tmp` (não usar em produção)
- A rotação de token remoto continua chegando por comando tipado enquanto o fluxo remoto ainda usa bootstrap/sync/ack.

## Build e deploy

```bash
# Build do agent-ui (Wails) — requer Wails CLI instalado
wails build

# Build do agent-service
go build ./cmd/agent-service/...

# Installer Windows
# Compilar com InnoSetup: deploy/windows-installer/AgenteTrilink.iss
```

**Dependências de runtime (incluídas no installer):**
- WebView2 Runtime (para Wails)
- RustDesk client
- gbak (Firebird backup tool)
