# API — Módulo rotinas-mensais

> `apps/api/src/modules/rotinas-mensais/` · Atualizado em: 2026-05-15

---

## Responsabilidade

O módulo `rotinas-mensais` gerencia o fluxo mensal recorrente de solicitação de documentação contábil a clientes. Para cada empresa configurada, gera competências (instâncias mensais), permite o envio de solicitações via WhatsApp e rastreia todo o ciclo até a conclusão.

## Procedures tRPC expostas (`trpc.rotinasMensais.*`)

| Procedure              | Tipo     | Descrição                                                              |
|------------------------|----------|------------------------------------------------------------------------|
| `list`                 | query    | Lista empresas candidatas à rotina com status e sumário                |
| `getCompanyConfig`     | query    | Retorna configuração da empresa (cria defaults se não existir)         |
| `listCompetencies`     | query    | Lista competências (instâncias mensais) com filtros de status/mês/ano  |
| `upsertCompanyConfig`  | mutation | Cria ou atualiza configuração de rotina da empresa                     |
| `syncCompetencies`     | mutation | Sincroniza competências para um mês/ano específico                     |
| `sendManualRequest`    | mutation | Envia solicitação manual via WhatsApp e registra a tentativa           |
| `updateCompetencyStatus` | mutation | Atualiza status da competência com timestamps automáticos            |

## Status de candidatura (`MonthlyRoutineCandidateStatus`)

| Status                | Condição                                                    |
|-----------------------|-------------------------------------------------------------|
| `READY_TO_CONFIGURE`  | Tem escritório contábil e contato primário vinculados       |
| `NO_ACCOUNTING_FIRM`  | Sem escritório contábil vinculado                           |
| `NO_PRIMARY_CONTACT`  | Tem escritório mas sem contato primário definido            |

---

## Workflow de status (`MonthlyRoutineExecutionStatus`)

```
PENDING → WAITING_CUSTOMER → RECEIVED → SENT_TO_ACCOUNTING → COMPLETED
   ↑                ↑
   └── OVERDUE ←────┘ (automático por vencimento ou timeout)
   └── CANCELED (manual)
```

| Status               | Descrição                                             |
|----------------------|-------------------------------------------------------|
| `PENDING`            | Aguardando envio de solicitação ao cliente            |
| `WAITING_CUSTOMER`   | Solicitação enviada, aguardando confirmação           |
| `RECEIVED`           | Cliente confirmou disponibilidade dos documentos      |
| `SENT_TO_ACCOUNTING` | Documentos enviados ao escritório contábil            |
| `COMPLETED`          | Ciclo concluído                                       |
| `OVERDUE`            | Passou do vencimento sem avançar (marcado auto)       |
| `CANCELED`           | Cancelado manualmente                                 |

---

## Timestamps automáticos por status

| Transição para          | Campo setado automaticamente |
|-------------------------|------------------------------|
| `WAITING_CUSTOMER`      | `requestedAt`                |
| `RECEIVED`              | `receivedAt`                 |
| `SENT_TO_ACCOUNTING`    | `sentAt`                     |
| `COMPLETED`             | `completedAt`                |

Os timestamps são setados apenas na **primeira** transição para aquele status (não sobrescreve se já preenchido).

## Configurações do módulo (`MonthlyRoutineModuleSettings`)

Persistidas em `SystemSetting` (chave a definir). **Nota:** em 2026-05-15 estas configurações estão definidas nos contratos mas ainda não são consumidas pelo service — o canal de envio é sempre WhatsApp direto via `EvolutionClient`.

| Campo                       | Padrão           | Descrição                                           |
|-----------------------------|------------------|-----------------------------------------------------|
| `defaultChannel`            | `CHATWOOT`       | Canal padrão de envio                               |
| `fallbackToDirectWhatsapp`  | `true`           | Usa WhatsApp direto se Chatwoot falhar              |
| `reuseOpenConversation`     | `true`           | Reaproveita conversa aberta no Chatwoot             |
| `markConversationAsPending` | `true`           | Marca conversa como pendente após envio             |
| `createPrivateNoteOnDispatch` | `true`         | Cria nota privada no Chatwoot ao disparar           |
| `applyRoutineLabels`        | `true`           | Aplica labels de rotina na conversa                 |
| `routineLabel`              | `"rotina_mensal"` | Label aplicada                                     |
| `competencyLabelPrefix`     | `"competencia"`  | Prefixo de label de competência                     |

## Fluxo de `ensureCompetenciesForScope`

Chamado inline em `list()`, `listCompetencies()` e `upsertCompanyConfig()`.

```
1. Busca todos os configs ativos no escopo
2. Para cada config:
   a. Calcula dueDate = min(config.dueDay, lastDayOfMonth) UTC 12:00
   b. findUnique competência pelo índice (configId, year, month)
   c. Se não existe: cria com status PENDING ou OVERDUE (se dueDate < now)
   d. Se existe e dueDate mudou: atualiza dueDate
3. Retorna { generated, updated }
```

> **Atenção:** Esta operação executa queries individuais em loop sequencial. Com volume alto de empresas, é candidata a otimização via batch ou job periódico.

---

## Fluxo de `markOverdueCompetencies`

Chamado inline em `listCompetencies()` antes de retornar os dados.

```
1. Lê AutomationModuleSettings (waitingCustomerTimeoutEnabled, waitingCustomerTimeoutHours)
2. Filtra registros candidatos a OVERDUE:
   - PENDING com dueDate < now
   - WAITING_CUSTOMER com updatedAt <= (now - timeoutHours) [se timeout ativo]
3. Para cada candidato:
   a. update status = OVERDUE
   b. createHistoryEntry (type: AUTO_OVERDUE)
   c. Se era WAITING_CUSTOMER: automationWhatsappService.sendMonthlyRoutineOverdueNotification
```

> **Atenção:** Executa updates individuais em loop síncrono. Candidato a `updateMany` + execução paralela de notificações.
