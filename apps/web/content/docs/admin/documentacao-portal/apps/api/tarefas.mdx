## Responsabilidade

O modulo `tarefas` centraliza dois tipos de demanda operacional:

| Tipo | Descricao |
|------|-----------|
| `ROTINA_MENSAL` | Instancia recorrente gerada por mes para empresas com configuracao ativa. |
| `TAREFA` | Demanda avulsa criada manualmente ou automaticamente a partir de um ticket. |

Ambos compartilham:
- workflow de status
- historico de auditoria
- solicitacoes manuais via WhatsApp
- listagem unificada no portal

## Procedures tRPC

As procedures expostas em `trpc.tarefas.*` sao:

| Procedure | Tipo | Descricao |
|-----------|------|-----------|
| `list` | query | Lista empresas candidatas a configuracao de rotina mensal. |
| `listTasks` | query | Lista tarefas com filtros por tipo, status, busca, mes e ano. |
| `getTask` | query | Retorna o detalhe completo da tarefa. |
| `getCompanyConfig` | query | Retorna a configuracao mensal da empresa. |
| `upsertCompanyConfig` | mutation | Cria ou atualiza `task_config`. |
| `createTask` | mutation | Cria tarefa avulsa manualmente. |
| `syncCompetencies` | mutation | Gera ou reconcilia tarefas mensais para um recorte de mes/ano. |
| `sendManualRequest` | mutation | Dispara solicitacao manual via WhatsApp. |
| `updateTaskStatus` | mutation | Atualiza status e registra historico. |

## Modelos de banco

| Tabela | Papel |
|--------|-------|
| `task_config` | Configuracao mensal por empresa. |
| `task` | Registro principal de rotina mensal ou tarefa avulsa. |
| `task_request` | Historico de tentativas de envio manual. |
| `task_history` | Auditoria de transicoes e eventos. |

### Relacionamentos

```text
Company
|-- task_config (0..1)
|   `-- task (N)          -> rotinas mensais geradas
`-- task (N)              -> tarefas avulsas ou mensais
    |-- task_request (N)
    |-- task_history (N)
    |-- ticket?           -> Conversation
    `-- assignedTo?       -> User
```

## Modelo `task`

Campos mais relevantes:

| Campo | Tipo | Observacao |
|-------|------|------------|
| `id` | string | Identificador da tarefa |
| `type` | enum | `ROTINA_MENSAL` ou `TAREFA` |
| `companyId` | string | Empresa vinculada |
| `configId` | string? | Preenchido para rotina mensal |
| `title` | string | Titulo exibido na UI |
| `description` | string? | Texto livre |
| `year` | int? | Mes/ano da rotina mensal |
| `month` | int? | Mes/ano da rotina mensal |
| `status` | enum | Estado operacional atual |
| `dueDate` | DateTime | Prazo principal |
| `ticketId` | string? | Origem em ticket, quando existir |
| `assignedToId` | string? | Responsavel |
| `notes` | string? | Observacoes internas |

> A rotina mensal usa `year` e `month`. O modelo atual nao usa os nomes antigos `competencyYear` ou `competencyMonth`.

## Workflow de status

```text
PENDING -> WAITING_CUSTOMER -> RECEIVED -> SENT_TO_ACCOUNTING -> COMPLETED
   |              |
   |              `-> OVERDUE
   `-> CANCELED
```

| Status | Significado |
|--------|-------------|
| `PENDING` | Ainda nao houve acao inicial |
| `WAITING_CUSTOMER` | Solicitação enviada ao cliente |
| `RECEIVED` | Cliente respondeu ou entregou o material |
| `SENT_TO_ACCOUNTING` | Material encaminhado ao escritorio |
| `COMPLETED` | Fluxo encerrado |
| `OVERDUE` | Prazo vencido sem avancar |
| `CANCELED` | Encerrado manualmente |

### Timestamps automaticos

| Status de destino | Campo preenchido |
|-------------------|------------------|
| `WAITING_CUSTOMER` | `requestedAt` |
| `RECEIVED` | `receivedAt` |
| `SENT_TO_ACCOUNTING` | `sentAt` |
| `COMPLETED` | `completedAt` |

## Integracao com tickets

Quando um ticket elegivel e resolvido, o modulo pode criar uma tarefa avulsa automaticamente.

Campos globais usados pela geracao:

| Campo | Tipo | Uso |
|-------|------|-----|
| `autoCreateOnTicketResolved` | boolean | Liga a geracao automatica |
| `autoTaskDueDays` | int | Prazo padrao em dias |
| `autoTaskTitle` | string? | Template do titulo |
| `autoTaskAssignToTicketOwner` | boolean | Atribui ao responsavel do ticket |

## UI do portal

Rota principal:

```text
/portal/tarefas
```

Comportamento atual da tela:
- busca por texto livre
- filtro por status
- filtro por tipo (`ROTINA_MENSAL` ou `TAREFA`)
- recorte mensal padrao pelo mes corrente
- paginacao server-driven
- dialogos para detalhe, disparo manual e atualizacao de status

O modulo de configuracao global aparece em:

```text
/portal/configuracoes?tab=tarefas
```

## Job periodico

`tarefas-job.service.ts` agenda a execucao recorrente do modulo. Na pratica, ele delega para o service principal tarefas como:
- garantir geracao de rotinas faltantes
- marcar itens vencidos
- processar automacoes e lembretes dependentes de configuracao

## Migracao do legado

Historicamente o modulo veio de `rotinas-mensais`, mas o estado atual esperado e:

| Legado | Atual |
|--------|-------|
| `monthly_routine_config` | `task_config` |
| `monthly_routine_competency` | `task` |
| `monthly_routine_request` | `task_request` |
| `monthly_routine_history` | `task_history` |
| `trpc.rotinasMensais.*` | `trpc.tarefas.*` |
| `/settings/monthly-routines` | `/settings/tarefas` |

<Callout type="warn">
  Se algum ambiente ainda estiver usando paths ou nomes antigos, trate isso como debito tecnico de migracao e nao como superficie oficial do modulo.
</Callout>

## Pontos de atencao

- O modulo usa o mesmo dominio para recorrencia mensal e tarefa avulsa, entao a UI deve sempre exibir `type`.
- A listagem precisa refletir `pagination`, `status` e `type` na URL para manter navegacao consistente.
- Permissoes continuam centralizadas em `tarefas:view`, `tarefas:view_all` e `tarefas:manage`.
