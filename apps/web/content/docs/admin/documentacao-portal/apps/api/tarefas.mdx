## Responsabilidade

O módulo `tarefas` centraliza dois tipos de demandas operacionais:

| Tipo | Descrição |
|------|-----------|
| **`ROTINA_MENSAL`** | Fluxo mensal recorrente de solicitação de documentação contábil. Gerado automaticamente por job periódico para cada empresa com rotina ativa. |
| **`TAREFA`** | Demanda avulsa — criada manualmente por um colaborador ou gerada automaticamente ao fechar um ticket. Sem vínculo obrigatório a mês/ano. |

Ambos os tipos compartilham o mesmo workflow de status, histórico de auditoria e integração com WhatsApp.

## Modelos de banco de dados

```
task_config              ← configuração de rotina mensal por empresa (1 por empresa)
task                     ← tarefa ou instância de rotina mensal (N por empresa)
task_request             ← tentativas de envio WhatsApp (N por tarefa)
task_history             ← auditoria de mudanças (N por tarefa)
```

**Relacionamentos:**
```
Company (1)
  ├── task_config (0..1)                  ← só para ROTINA_MENSAL
  │     └── task (N)                      ← competências mensais geradas
  └── task (N)                            ← tarefas avulsas (TAREFA)
        ├── task_request (N)              ← tentativas de envio
        ├── task_history (N)             ← auditoria
        ├── ticket? (FK → Conversation)  ← origem (se gerada por ticket)
        └── assignedUser? (FK → User)    ← responsável
```

## Tipo de tarefa

### `ROTINA_MENSAL`

- Gerada automaticamente pelo job periódico para cada empresa com `task_config.isActive = true`
- Vinculada obrigatoriamente a `competencyYear` + `competencyMonth`
- Prazo calculado a partir de `task_config.dueDay`
- Uma por empresa por mês

### `TAREFA`

- Criada manualmente via `createTask` ou automaticamente via integração com tickets
- `dueDate` livre, definido no momento de criação
- Pode ou não ter `ticketId` (se gerada a partir de fechamento de ticket)
- Pode ter responsável (`assignedToId`)
- Sem vínculo com `task_config`

## Workflow de status (`TaskExecutionStatus`)

```
PENDING → WAITING_CUSTOMER → RECEIVED → SENT_TO_ACCOUNTING → COMPLETED
   ↑
   └── OVERDUE (automático por vencimento)
   └── CANCELED (manual)
```

| Status | Descrição |
|--------|-----------|
| `PENDING` | Aguardando ação inicial |
| `WAITING_CUSTOMER` | Solicitação enviada, aguardando resposta do cliente |
| `RECEIVED` | Cliente confirmou ou entregou o necessário |
| `SENT_TO_ACCOUNTING` | Enviado ao escritório contábil (específico de rotinas) |
| `COMPLETED` | Tarefa concluída |
| `OVERDUE` | Passou do prazo sem avançar (marcado automaticamente) |
| `CANCELED` | Cancelado manualmente |

### Timestamps automáticos por transição

| Transição para | Campo setado |
|----------------|--------------|
| `WAITING_CUSTOMER` | `requestedAt` |
| `RECEIVED` | `receivedAt` |
| `SENT_TO_ACCOUNTING` | `sentAt` |
| `COMPLETED` | `completedAt` |

Timestamps são setados apenas na **primeira** transição — não sobrescreve se já preenchido.

## Integração com Tickets

Quando um ticket muda para status `RESOLVED`, o módulo `tarefas` pode gerar automaticamente uma tarefa para a empresa vinculada ao ticket.

### Fluxo

```
tickets.service.ts: updateTicket()
  └── status → RESOLVED
        └── tarefas.service.ts: createFromTicket(ticket)
              ├── verifica se ticket tem companyId
              ├── verifica configuração: geração automática habilitada?
              └── cria task {
                    type: TAREFA,
                    companyId: ticket.companyId,
                    ticketId: ticket.id,
                    title: `Acompanhamento pós-atendimento — ${ticket.subject}`,
                    dueDate: now + configuredDays,
                    status: PENDING,
                    createdById: 'system'
                  }
```

### Configuração

A geração automática por ticket é controlada por configuração global do módulo:

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `autoCreateOnTicketResolved` | boolean | Habilita criação automática ao resolver ticket |
| `autoTaskDueDays` | int | Dias de prazo para tarefa gerada (padrão: 3) |
| `autoTaskTitle` | string? | Template do título (suporta `{ticket_subject}`, `{company_name}`) |
| `autoTaskAssignToTicketOwner` | boolean | Atribui ao responsável do ticket automaticamente |

## Configuração de rotina mensal por empresa (`task_config`)

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `isActive` | boolean | Habilita geração de rotinas mensais para esta empresa |
| `title` | string (3–120) | Título da rotina exibido nas mensagens |
| `dueDay` | int (1–31) | Dia do mês de vencimento (auto-ajusta para último dia do mês) |
| `reminderDays` | int (0–30) | Dias antes do vencimento para lembrete automático |
| `clientContactId` | string? | Contato do cliente para solicitações |
| `accountingContactId` | string? | Contato do escritório contábil para entrega |
| `requiredDocuments` | string[] | Lista de documentos esperados |
| `notes` | string? | Observações internas |

**Documentos padrão:**
`Sintegra`, `Livro de entrada`, `Livro de saida`, `Relatorio de PIS e COFINS`,
`SPED Fiscal`, `SPED Contribuicoes`, `XML NF-e entrada`, `XML NF-e saida`, `XML NF-e CT-e`

## Templates de mensagem WhatsApp

| Template | Uso sugerido |
|----------|-------------|
| `REQUEST_CONFIRMATION` | Solicitação inicial |
| `FIRST_REMINDER` | Primeiro lembrete |
| `SECOND_REMINDER` | Segundo lembrete |

Aceita mensagem customizada (campo `message`, 8–4.000 caracteres). Se nenhum template nem mensagem for fornecida, usa `REQUEST_CONFIRMATION` como padrão.

## Migração do módulo `rotinas-mensais`

<Callout type="warn">
  O módulo `rotinas-mensais` será descontinuado e substituído por `tarefas`. As tabelas do banco serão renomeadas e os dados migrados conforme plano abaixo.
</Callout>

### Renomeação de tabelas

| Tabela antiga | Tabela nova | Observação |
|--------------|-------------|------------|
| `monthly_routine_config` | `task_config` | Mantém todos os campos |
| `monthly_routine_competency` | `task` | Adiciona campos: `type`, `ticketId`, `assignedToId`, `description` |
| `monthly_routine_request` | `task_request` | Mantém todos os campos |
| `monthly_routine_history` | `task_history` | Mantém todos os campos |

### Passos de migração (Prisma)

```sql
-- 1. Renomear tabelas
ALTER TABLE monthly_routine_config RENAME TO task_config;
ALTER TABLE monthly_routine_competency RENAME TO task;
ALTER TABLE monthly_routine_request RENAME TO task_request;
ALTER TABLE monthly_routine_history RENAME TO task_history;

-- 2. Adicionar campos novos em `task`
ALTER TABLE task ADD COLUMN type TEXT NOT NULL DEFAULT 'ROTINA_MENSAL';
ALTER TABLE task ADD COLUMN ticket_id TEXT REFERENCES "Conversation"(id);
ALTER TABLE task ADD COLUMN assigned_to_id TEXT REFERENCES "User"(id);
ALTER TABLE task ADD COLUMN description TEXT;
ALTER TABLE task ADD COLUMN due_date TIMESTAMP;

-- 3. Tornar campos de competência opcionais
ALTER TABLE task ALTER COLUMN competency_year DROP NOT NULL;
ALTER TABLE task ALTER COLUMN competency_month DROP NOT NULL;
ALTER TABLE task ALTER COLUMN config_id DROP NOT NULL;
```

### Renomeação de procedures tRPC

| Procedure antiga | Procedure nova |
|-----------------|---------------|
| `trpc.rotinasMensais.list` | `trpc.tarefas.list` |
| `trpc.rotinasMensais.listCompetencies` | `trpc.tarefas.listTasks` |
| `trpc.rotinasMensais.getCompetency` | `trpc.tarefas.getTask` |
| `trpc.rotinasMensais.getCompanyConfig` | `trpc.tarefas.getCompanyConfig` |
| `trpc.rotinasMensais.upsertCompanyConfig` | `trpc.tarefas.upsertCompanyConfig` |
