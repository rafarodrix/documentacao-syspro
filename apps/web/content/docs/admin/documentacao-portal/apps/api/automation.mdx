## Responsabilidade

O módulo `automation` gerencia **automações de notificação via WhatsApp** disparadas por eventos do portal. Envia mensagens para grupos WhatsApp configurados quando tickets são criados, transferidos, têm status alterado, releases são publicadas ou a SEFAZ cai/se recupera.

---

## Arquivos

```
src/modules/automation/
├── automation.module.ts             ← módulo NestJS
├── automation-settings.service.ts  ← leitura e merge de configurações
└── automation-whatsapp.service.ts  ← envio de notificações via Evolution
```

## AutomationSettingsService

Lê e persiste as configurações do módulo de automação no banco (`SystemSetting`).

**Chaves de configuração no banco:**

| Chave                          | Conteúdo                                     |
|-------------------------------|----------------------------------------------|
| `automation.module.settings`  | `AutomationModuleSettings` (JSON serializado)|
| `tickets.module.settings`     | `TicketModuleSettings` (JSON serializado)    |

**Métodos principais:**

```typescript
// Lê as configurações de automação (com fallback para defaults)
readAutomationModuleSettings(): Promise<AutomationModuleSettings>

// Lê configurações de tickets já mergeadas com automation
readMergedTicketModuleSettings(): Promise<TicketModuleSettings>

// Merge manual: injeta grupos de notificação do automation no TicketModuleSettings
mergeAutomationSettingsIntoTicketSettings(
  settings: TicketModuleSettings,
  automationSettings: AutomationModuleSettings
): TicketModuleSettings
```

O merge injeta os grupos de notificação do `automation` dentro do `TicketModuleSettings`, para que o módulo de tickets não precise conhecer o módulo de automação diretamente ao exibir configurações.

## AutomationWhatsappService

Responsável por enviar mensagens de notificação para grupos WhatsApp via Evolution API.

**Método de disparo por evento:**

| Método                                    | Quando é chamado                            |
|-------------------------------------------|---------------------------------------------|
| `sendTicketCreatedGroupNotification`      | Ticket novo criado (suporte ou desenvolvimento) |
| `sendTicketTeamRoutingGroupNotifications` | Ticket transferido entre equipes (SUPORTE ↔ DESENVOLVIMENTO) |
| `sendTicketStatusGroupNotification`       | Ticket movido para TESTING ou retornado de TESTING |
| `sendReleasePublishedNotification`        | Release publicada                           |
| `sendSefazRouteStatusNotification`        | Rota SEFAZ caiu ou se recuperou (individual)|
| `sendSefazRouteStatusDigestNotification`  | Múltiplas rotas SEFAZ caíram/recuperaram (digest) |

## Eventos disponíveis (`WhatsAppAutomationEvent`)

```typescript
type WhatsAppAutomationEvent =
  | 'ticket_created_support'             // ticket criado para equipe Suporte
  | 'ticket_created_development'         // ticket criado para equipe Desenvolvimento
  | 'ticket_team_transfer_from_support'  // ticket saiu do Suporte
  | 'ticket_team_transfer_to_support'    // ticket chegou ao Suporte
  | 'ticket_team_transfer_from_development' // ticket saiu do Desenvolvimento
  | 'ticket_team_transfer_to_development'   // ticket chegou ao Desenvolvimento
  | 'ticket_status_testing'             // ticket entrou em Testes
  | 'ticket_status_testing_failed'      // ticket retornou de Testes (falhou)
  | 'release_published'                 // release publicada
  | 'sefaz_route_down'                  // rota SEFAZ indisponível
  | 'sefaz_route_recovered'             // rota SEFAZ normalizada
```

## Configuração (AutomationModuleSettings)

Persistida em `SystemSetting` com chave `automation.module.settings`:

```typescript
interface AutomationModuleSettings {
  autoAssignToCreator: boolean        // auto-atribui ticket ao criador
  autoResponseEnabled: boolean        // resposta automática ao abrir ticket
  autoResponseMessage: string         // mensagem da resposta automática
  requireTestingReturnReason: boolean // exige motivo ao retornar de testes
  whatsapp: {
    bindings: WhatsAppAutomationBinding[]  // grupos configurados
  }
}

interface WhatsAppAutomationBinding {
  id: string
  label: string      // nome amigável do grupo (ex: "Grupo Suporte")
  jid: string        // JID do grupo WhatsApp (ex: "5511...@g.us")
  active: boolean    // habilita/desabilita este binding
  automations: {     // quais eventos este grupo recebe
    ticketCreatedSupport: boolean
    ticketCreatedDevelopment: boolean
    ticketTeamTransferFromSupport: boolean
    ticketTeamTransferToSupport: boolean
    ticketTeamTransferFromDevelopment: boolean
    ticketTeamTransferToDevelopment: boolean
    ticketStatusTesting: boolean
    ticketStatusTestingFailed: boolean
    releasePublished: boolean
    sefazRouteDown: boolean
    sefazRouteRecovered: boolean
  }
}
```

**Defaults:** todos os `automations` flags são `false`. Nenhum grupo recebe notificações até ser explicitamente habilitado.

## Fluxo de envio

Para cada notificação, o serviço executa:

```
1. Lê AutomationModuleSettings do banco
2. Filtra bindings ativos que têm o evento habilitado → targetGroups
3. Se nenhum grupo: loga debug e retorna silenciosamente
4. resolveNotificationContext():
   ├── busca conexão Evolution da empresa do ticket (companyId)
   ├── aplica score: empresa match (+100), grupo permitido (+20), source=database (+10)
   ├── fallback: contexto padrão configurado
   └── fallback final: qualquer conexão ativa
5. Se nenhuma conexão: loga warn + registra nota de falha no ticket
6. Monta mensagem formatada em Markdown WhatsApp (*negrito*, `código`)
7. dispatchToGroups(): envia para cada grupo com retry (3x, delay 1.5s)
8. Se todos falharam: registra nota de falha no ticket
```

## Retry de envio

O `sendTextMessageWithRetry` usa 3 tentativas com delay progressivo:

| Tentativa | Delay antes |
|-----------|-------------|
| 1ª        | imediato    |
| 2ª        | 1.5s        |
| 3ª        | 3.0s        |

Erros classificados como **não retryáveis** (ex: grupo não encontrado, não autorizado) param imediatamente na 1ª falha.

**Classificação de erros (`classifyAutomationDispatchError`):**

| Tipo de erro                    | Kind                       | Retryable |
|---------------------------------|----------------------------|-----------|
| `failed to get group members`   | `group_metadata_timeout`   | ✓         |
| `info query timed out`          | `group_metadata_timeout`   | ✓         |
| `timed out` / `timeout`         | `provider_timeout`         | ✓         |
| `not-authorized` / `unauthorized` | `provider_unauthorized`  | ✗         |
| `not found` + `@g.us`           | `group_not_found`          | ✗         |
| outros                          | `provider_error`           | ✗         |

## Notas de falha no ticket

Quando uma notificação falha, o serviço registra uma nota interna no ticket como `ConversationMessage` com:
- `direction: INTERNAL`
- `type: SYSTEM_EVENT`
- `authorKind: EXTERNAL`

Exemplo de nota registrada:
```
[Automacao] Notificacao de abertura nao entregue a nenhum grupo do WhatsApp.
Ticket: #1234
Conexao: minha-instancia
Tentativas: 3
Falhas: 3
- Grupo Suporte (12345-67890@g.us): Timeout do provider ao tentar enviar...
```

## Resolução de contexto Evolution

`resolveNotificationContext` implementa um algoritmo de score para selecionar a melhor instância Evolution:

1. **Contextos da empresa** (se `companyId` informado): lista conexões ativas vinculadas à empresa
2. **Score**: empresa match = +100, grupo na allowlist = +20, source database = +10
3. **Fallback**: contexto padrão configurado (sem empresa específica)
4. **Fallback final**: qualquer conexão Evolution ativa

## URL do portal nas notificações

O serviço tenta montar URLs absolutas dos tickets/releases/SEFAZ nas mensagens. Precedência:

1. Header `x-portal-origin` da request
2. Headers `origin` ou `referer`
3. Variáveis de ambiente: `NEXT_PUBLIC_APP_URL`, `FRONTEND_URL`, `PORTAL_URL`, `APP_URL` (primeira válida)
4. Header `host` + protocolo inferido de `x-forwarded-proto`

**Normalização de hostname:** URLs com `backend.*` ou `api.*` têm o prefixo substituído por `ajuda.*` para apontar para o frontend.
