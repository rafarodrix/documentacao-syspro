# Guia: uso de DESIGN.md no projeto

O repositorio [`google-labs-code/design.md`](https://github.com/google-labs-code/design.md) define um formato aberto para descrever identidade visual e tokens de design para agentes de codigo.

No nosso projeto ele serve como um **contrato de design legivel por agentes**, e nao como fonte direta de build.

## Onde o arquivo deve ficar

O arquivo canonical fica em:

```text
apps/web/docs/design-system/DESIGN.md
```

Ele **nao** deve ficar dentro de `apps/web/content/docs`, porque o script `apps/web/scripts/docs-check.mjs` rejeita qualquer arquivo que nao seja `.mdx` ou `meta.json` nessa arvore.

## O que continua sendo fonte de verdade em runtime

Hoje a UI publicada continua derivando destes arquivos:

- `packages/ui/src/tokens.css`
- `apps/web/src/app/globals.css`

O `DESIGN.md` deve ser mantido sincronizado com eles, mas **nao substitui** esses arquivos neste passo.

## Fluxo recomendado

1. Ajuste tokens reais em `packages/ui/src/tokens.css`.
2. Se houver impacto semantico relevante, atualize `apps/web/docs/design-system/DESIGN.md`.
3. Use o `DESIGN.md` como contexto para agentes, prototipos e futuras telas.
4. Se quiser validar a estrutura do arquivo, rode a CLI do projeto do Google Labs manualmente.

## Comandos uteis

```bash
npx @google/design.md lint apps/web/docs/design-system/DESIGN.md
```

```bash
npx @google/design.md spec --rules
```

```bash
npx @google/design.md export --format css-tailwind apps/web/docs/design-system/DESIGN.md
```

No Windows, se o time decidir instalar a CLI em vez de usar `npx`, prefira o alias `designmd` em scripts, conforme a documentacao do projeto.

## Como usar no dia a dia

- Em prompts para agentes: informe que o agente deve ler `apps/web/docs/design-system/DESIGN.md` antes de propor UI nova.
- Em revisoes: compare a proposta com `tokens.css` e recuse qualquer divergencia que nao tenha sido intencional.
- Em novas interfaces: use o arquivo para explicar tom visual, contraste, tipografia e uso de componentes.

## Limites da adocao atual

- O arquivo ainda e documental; nao ha sincronizacao automatica para CSS.
- Como nossos tokens reais usam bastante `oklch(...)`, parte da representacao no `DESIGN.md` pode ser mais conceitual do que literal.
- Se o time quiser tornar `DESIGN.md` fonte primaria no futuro, vale discutir uma etapa separada de export/sync.
