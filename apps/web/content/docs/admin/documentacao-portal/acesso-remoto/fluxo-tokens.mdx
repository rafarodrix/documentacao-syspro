# Acesso Remoto - Tokens e Sessoes

> Ciclo de vida do agentToken, rotacao de credenciais e gerenciamento de sessoes. Atualizado em: 2026-07-13

---

## agentToken

O `agentToken` e a credencial principal usada pelo agente Go para autenticar
as chamadas remotas apos o bootstrap.

### Caracteristicas

- Formato visual `agt_<random-hex>`
- Gerado no backend
- TTL configuravel
- Persistido na maquina com DPAPI
- Vinculado a um unico `RemoteHost`
- Falha fechada no agent quando o contrato remoto de bootstrap/sync vier incompleto ou inesperado

### Ciclo de vida

1. O portal gera o `agentToken` no bootstrap.
2. O agente persiste o token com DPAPI.
3. O agente usa esse token nos ciclos de `sync` e `ack`.
4. Se o token expirar, o backend invalida a credencial e o host volta a exigir bootstrap autenticado.

### Rotacao manual

Administrador pode forcar rotacao via:

```text
POST /api/remote/hosts/:id/agent-token
```

## installToken

O `installToken` e usado apenas no bootstrap autenticado de hosts ja vinculados.

### Regras atuais

- E gerado e controlado pelo portal
- E entregue ao agente pelo `discover` somente quando o host ja esta vinculado e apto a bootstrap
- So e aceito quando houver `discover` recente e vinculado da mesma maquina
- E reemitido a cada tentativa de bootstrap autorizada pelo `discover`
- Tem vida curta
- E invalidado ao final de um bootstrap bem-sucedido
- Fica restrito a uma janela curta de autorizacao operacional
- Nao e mais configurado localmente no `.env` do agente

## discoveryToken

O `discoveryToken` identifica a maquina antes do bootstrap.

### Regras atuais

- E a unica credencial remota configurada localmente no agent
- Permite descoberta e triagem inicial
- Nao concede `sync` autenticado por si so
- O portal responde com o estado operacional do host e, quando o item ainda estiver em `pending_link`, ja libera bootstrap tecnico com `discoveryToken + discoveredHostId`
- O portal so informa `installToken` quando o host ja esta vinculado e apto a bootstrap autenticado
- O agent aceita apenas o contrato `discover.v2`; payload/flow inesperado entra em erro estruturado local

### Resultados comuns

- `pending_link`
- `linked_host_detected`
- `host_bootstrap_required`
- `token_invalid`

Quando um host/dispositivo e removido do portal, o `discover` deixa de rematerializar esse registro automaticamente. O retorno operacional permanece em modo de triagem ate nova autorizacao operacional ou reinstalacao controlada.

O `discover` tambem nao promove mais uma maquina para `linked_host_detected`
apenas porque encontrou um `RemoteHost` legado com o mesmo `rustdeskId`.
Bootstrap autenticado so volta a acontecer depois de um `RemoteDiscoveredHost`
explicito ser vinculado/autorizado no portal.

## Address Book Credentials

Credenciais para tecnicos configurarem o cliente RustDesk com o address book do portal.

| Escopo | Acesso | Quem pode criar |
| --- | --- | --- |
| `GLOBAL` | Todos os hosts de todas as empresas | ADMIN, SUPORTE |
| `COMPANY` | Apenas hosts de uma empresa especifica | ADMIN, CLIENTE_ADMIN |

### Endpoints

```text
GET  /api/remote/rustdesk/address-book/credentials
POST /api/remote/rustdesk/address-book/credentials
POST /api/remote/rustdesk/address-book/credentials/:id/rotate
POST /api/remote/rustdesk/address-book/credentials/:id/revoke
```

## Sessao remota

Estados operacionais:

1. `REQUESTED`
2. `STARTED`
3. `ENDED`

O cleanup de sessoes presas continua sendo feito pelo backend.

---

## Fila de comandos do agente

O portal pode devolver comandos no `sync`:

- `REAPPLY_ALIAS`
- `REAPPLY_CONFIG`
- `UPGRADE_CLIENT`
- `ROTATE_TOKEN_REQUIRED`

O agente executa e confirma por:

```text
POST /api/remote/rustdesk/ack
```

Quando o POST de ACK falha, a confirmacao entra em fila local persistida com:

- `agent_token` protegido em disco
- deduplicacao por `commandId`
- `next_attempt_at`
- `last_error_code`
- `last_error_message`
- backoff entre tentativas

---

## Observacao importante

Se um host estiver vinculado mas o portal ainda nao devolver `installToken`,
o agente permanece aguardando bootstrap autenticado. Ele nao usa mais
`REMOTE_INSTALL_TOKEN` local legado para tentar completar o onboarding.

Se o host ainda nao estiver vinculado, isso nao bloqueia mais a instalacao
tecnica do remoto: o agent pode concluir bootstrap/configuracao local e ficar
em estado "pronto para vinculo" ate a associacao empresarial no portal.
