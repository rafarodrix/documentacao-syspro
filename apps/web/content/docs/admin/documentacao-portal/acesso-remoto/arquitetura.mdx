# Acesso Remoto - Arquitetura Completa

> Visao end-to-end do modulo de acesso remoto (RustDesk + Agente + Portal).
> Atualizado em: 2026-07-17

---

## Visao geral do modulo

O modulo de acesso remoto permite que tecnicos Trilink acessem remotamente maquinas Windows de clientes de forma segura e auditada.

**Componentes:**

```text
Portal (apps/web + apps/api)
  |- Cadastro e gerenciamento de hosts
  |- Gerenciamento de sessoes e credenciais
  |- Address book para o cliente RustDesk
  `- Auditoria e logs de acesso

apps/api/modules/remote-admin
  |- REST endpoints do agente (heartbeat, sync, ack, bootstrap, discover)
  |- REST endpoints administrativos (hosts, sessoes, address book)
  `- Orquestra use cases do @dosc-syspro/remote-domain

packages/features/remote/domain
  `- Logica de negocio pura (sem deps externas)

packages/features/remote/infra
  `- Implementacoes das ports (Prisma, HTTP)

apps/agent (Go)
  |- Mantem a identidade base da instalacao
  |- Gerencia a capability remota do RustDesk
  `- Executa reconcile, sync e comandos do portal

RustDesk (open-source)
  |- Cliente: tecnico usa para se conectar
  `- Servidor relay: retransmite a conexao
```

| Metodo | Endpoint                         | Uso                          |

```text
Remote Capability
  ├── RemoteDiscoveredHost
  └── RemoteHost
```

## Estado atual do banco

O split principal do banco já está ativo:

```text
Device
  └── 1:N AgentInstallation
        └── 1:N AgentCapability
```

No fluxo atual:

- `Device` representa a máquina física
- `AgentInstallation` representa a instalação local e seu heartbeat base
- `AgentCapability` representa a capability remota associada ao `RemoteHost`
- `RemoteDiscoveredHost` para triagem remota
- `RemoteHost` para estado operacional do acesso remoto

Não existe mais estrutura paralela para presença base do agent; o vínculo operacional fica em `Device`, `AgentInstallation` e `AgentCapability`.

## Fluxo completo: tecnico acessa maquina do cliente

```text
1. Tecnico abre /portal/infraestrutura/hosts/:id
   |- Ve status ONLINE e botao "Iniciar Sessao"
   `- Clica em "Iniciar Sessao"

2. POST /api/remote/sessions
   <- { sessionId, rustdeskId: "123456789", credentials }
   Sessao criada com status REQUESTED

3. Tecnico abre o cliente RustDesk com as credenciais
   `- Digita rustdeskId "123456789" ou usa o address book

4. RustDesk conecta via relay.trilink.com.br

5. A capability remota do host continua sincronizada pelo `agent-service`
   `- o portal ja conhece o estado do host via `sync`

6. Portal atualiza sessao para STARTED

7. Tecnico faz o suporte

8. Tecnico encerra no portal ou fecha o RustDesk
   `- POST /api/remote/sessions/:id/stop
   Sessao -> ENDED
   Se ticketId: nota adicionada ao ticket automaticamente
```

## Seguranca e isolamento

- Cada usuario do portal so acessa hosts das empresas as quais tem permissao.
- `resolveScopedCompanyContext` valida o escopo em toda operacao administrativa.
- Rate limit em discovery (5 req/min por IP) previne enumeracao.
- `agentToken` continua sendo credencial da capability remota, nao da instalacao inteira.
- Credenciais de address book tem escopo (GLOBAL ou COMPANY) e podem ser revogadas individualmente.
- Todas as sessoes sao auditadas com inicio, fim, tecnico e duracao.
