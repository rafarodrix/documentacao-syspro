## Topologia atual

| Camada | Serviço | Provedor |
|--------|---------|----------|
| Frontend | `apps/web` (Next.js) | Vercel |
| Backend | `apps/api` (NestJS) | Railway |
| Servidor base (VPS) | Ubuntu 24.04 LTS | Contabo |
| Orquestração de containers | Dokploy | Contabo VPS |
| Reverse proxy / SSL | Traefik v3 | Contabo VPS (via Dokploy) |
| RustDesk Server | container Docker | Contabo VPS |
| Evolution Go | container Docker | Contabo VPS |
| Banco do portal | Prisma + PostgreSQL | Railway / Neon |
| Bancos Evolution Go | `evogo_auth`, `evogo_users` | Neon |

## Mapeamento de domínios

| Subdomínio | Serviço | Host |
|------------|---------|------|
| `api.<dominio>` | Evolution Go | VPS Contabo |
| `acesso.<dominio>` | RustDesk | VPS Contabo |
| `backend.<dominio>` | NestJS `apps/api` | Railway |

<Callout type="warn" title="Separação obrigatória">
  Não reutilizar `api.<dominio>` (Evolution) para o backend NestJS. O `apps/web` deve apontar para `backend.<dominio>/api`.
</Callout>

---

## Escopo por camada

### 1. Servidor (host)

- Provedor: Contabo VPS
- Sistema operacional: Ubuntu 24.04 LTS
- Dokploy instalado no host como plataforma de orquestração
- Traefik v3 gerenciado pelo Dokploy como reverse proxy central

### 2. Rede e segurança

- Traefik v3 recebe tráfego de internet (80/443) via entrypoints `web` e `websecure`
- SSL/TLS automatizado por Let's Encrypt (ACME HTTP Challenge)
- Roteamento por labels Docker/Swarm:
  - `api.<dominio>` → Evolution Go (container interno)
  - `acesso.<dominio>` → RustDesk (container)

### 3. Aplicação (containers)

- Evolution Go (v2) e RustDesk em Docker, orquestrados pelo Dokploy
- Configuração por `.env` com chaves e conexões externas
- Persistência fora do container (Neon), permitindo recriação segura do serviço
- Rede padrão: `dokploy-network`

### 4. Dados

- Provider: Neon (PostgreSQL serverless)
- `AUTH_DB` → `evogo_auth`
- `USERS_DB` → `evogo_users`

---

## Operação do proxy (Traefik)

Dashboard Traefik: `:8081` (acesso interno)

Regras obrigatórias:
- Cada serviço deve declarar explicitamente `traefik.http.routers.<nome>.service=<nome>-svc`
- Nunca depender de inferência automática do Traefik (causa erro `too many services`)
- **Não expor** `:8080` da Evolution diretamente para a internet
- Todos os containers devem estar na rede `dokploy-network`

<Callout type="info" title="Consulte o guia de VPS">
  Para detalhes de labels, Docker Swarm vs Compose e diagnóstico do Traefik, veja [Infraestrutura VPS](/portal/docs/admin/infraestrutura/infraestrutura-vps).
</Callout>

## Variáveis críticas por runtime

### Vercel (`apps/web`)

```bash
APP_BACKEND_API_URL=https://backend.<dominio>/api
APP_API_URL=https://backend.<dominio>/api
INTERNAL_API_KEY=<chave-compartilhada-com-api>
```

<Callout type="info" title="Legado">
  `APP_API_URL` ainda é suportado, mas `APP_BACKEND_API_URL` deve ser preferido em novos deployments.
</Callout>

### Railway (`apps/api`)

```bash
DATABASE_URL=
INTERNAL_API_KEY=
EVOLUTION_API_URL=https://api.<dominio>
EVOLUTION_API_KEY=
EVOLUTION_INSTANCE=
EVOLUTION_WEBHOOK_SECRET=
```

### Evolution Go (VPS)

```bash
SERVER_URL=https://api.<dominio>
GLOBAL_API_KEY=
POSTGRES_AUTH_DB=...sslmode=require
POSTGRES_USERS_DB=...sslmode=require
```

## Regras de integração

1. Web chama somente backend NestJS (`backend.<dominio>/api`).
2. Backend NestJS chama Evolution Go quando necessário (`api.<dominio>`).
3. Evolution envia webhook diretamente para o backend NestJS.
4. RustDesk roteado por subdomínio dedicado `acesso.<dominio>`.

---

## Fluxo de comunicação

<Steps>
  <Step>
    ### Cliente → Backend NestJS

    O browser (ou Server Action do `apps/web`) chama `https://backend.<dominio>/api` no Railway.
  </Step>
  <Step>
    ### Backend → Evolution Go

    Quando necessário, o NestJS chama `https://api.<dominio>`. O Traefik valida TLS e roteia para o container Evolution na rede `dokploy-network`.
  </Step>
  <Step>
    ### Evolution → Neon

    O container Evolution processa a requisição e persiste eventos/estado nos bancos Neon (`evogo_auth`, `evogo_users`).
  </Step>
  <Step>
    ### Evolution → Webhook NestJS

    Em eventos inbound (mensagens, status), a Evolution publica webhook diretamente para o endpoint do backend NestJS no Railway.
  </Step>
</Steps>

---

## Controles de segurança

- Evolution sem exposição pública direta (acesso via Traefik com TLS)
- TLS na borda com certificado automatizado (Let's Encrypt)
- `apikey` e segredos de ambiente para integração entre serviços
- CORS restrito apenas para origens autorizadas

---

## Manutenção recomendada

- Aplicar reboot da VPS após atualização de kernel/segurança
- Backup periódico dos diretórios operacionais via Dokploy
- Validar rotação de chaves (`GLOBAL_API_KEY`, `INTERNAL_API_KEY`, `EVOLUTION_WEBHOOK_SECRET`) em janela controlada
