## Objetivo

Concentrar apenas o backlog ainda nao concluido, priorizando:

- escalabilidade da arquitetura
- reducao de acoplamento
- qualidade de entrega
- preparacao para mobile e backend separado

## Diretriz de execucao

O backlog passa a seguir esta ordem:

1. consolidar o **modular monolith**
2. estabilizar contratos e boundaries
3. extrair o **monorepo estrutural**
4. extrair o **monorepo de dominio**
5. abrir caminho para mobile e API dedicada

Nao vamos tratar workspaces, mobile e API separada como primeiros passos.

## Fase A - Modular Monolith Disciplinado

### EPIC A1 - Migracao por feature para `src/features`

- [ ] `INFRA-001` [Alta] Migrar `company` para `src/features/company`.
Criterios de aceite:
- `domain`, `application`, `interface` criados.
- `page.tsx` e `actions` consumindo a feature, nao regra espalhada.
Status atual:
- application e interface iniciados em `src/features/company`.
- pages e componentes principais de empresa ja consomem a feature.
- falta mover a implementacao de `src/actions/platform/company-actions.ts` para `application`.

- [ ] `INFRA-002` [Alta] Migrar `tax` para `src/features/tax`.
Criterios de aceite:
- consultas, sync e UI separados por camada.
- regras de negocio fora de page/component.
Status atual:
- application e interface iniciados em `src/features/tax`.
- pages e rota de sync ja consomem a feature.
- falta extrair a implementacao de `src/actions/tax/tax-actions.ts`.

- [ ] `INFRA-003` [Alta] Migrar `settings/contracts` para `src/features`.
Criterios de aceite:
- boundaries claros por modulo.
- sem acesso direto a infraestrutura na interface.
Status atual:
- settings e contracts ja possuem `application` e `interface`.
- `src/actions/platform/settings-actions.ts` e `src/actions/platform/contract-actions.ts` agora atuam como ponte fina.
- falta concluir `queries/domain` e expandir o padrao para o restante do modulo administrativo.

- [ ] `INFRA-004` [Alta] Migrar `users/authz` para `src/features/user-access`.
Criterios de aceite:
- fluxo de cadastro, memberships e roles separados por feature.
- sem regra critica de permissao espalhada em page/component.

### EPIC A2 - Actions finas e contratos internos

- [ ] `INFRA-005` [Alta] Transformar `src/actions/*` em camada fina de transporte.
Criterios de aceite:
- `actions` delegam para `application`.
- sem duplicacao de query, filtro ou regra em action.
Status atual:
- concluido para `tickets`, `settings` e `contracts`.
- em andamento para `company` e `tax`.

- [ ] `INFRA-006` [Alta] Criar view models e contratos por feature dentro do app atual.
Criterios de aceite:
- dashboard, listagem e detalhes usam o mesmo contrato.
- nenhum componente forte depende de shape improvisado local.

- [ ] `INFRA-007` [Alta] Padronizar gateways/adapters tecnicos em `core/infrastructure`.
Criterios de aceite:
- Zammad, company registry, tax sync e observabilidade seguem contrato explicito.
- pagina nao fala com client tecnico direto.

### EPIC A3 - Guardrails e governanca

- [ ] `INFRA-008` [Alta] Criar checklist de PR para arquitetura por feature.
Criterios de aceite:
- revisar camada, acoplamento, contrato e regra duplicada.

- [ ] `INFRA-009` [Alta] Adicionar lint arquitetural basico por convencao.
Criterios de aceite:
- impedir import proibido de `interface -> prisma/lib`.
- impedir page/component chamando infraestrutura diretamente onde ja houver feature.

## Fase B - Monorepo Estrutural

### EPIC B1 - Workspace e shell

- [ ] `INFRA-010` [Alta] Estruturar raiz do monorepo com `apps/web` e `packages/*`.
Criterios de aceite:
- `pnpm-workspace.yaml` e `turbo.json` configurados.
- `apps/web` criado sem regressao funcional.

- [ ] `INFRA-011` [Alta] Mover a aplicacao Next.js para `apps/web`.
Criterios de aceite:
- build, docs, auth e app funcionando a partir de `apps/web`.
- imports ajustados sem alias quebrado.

- [ ] `INFRA-012` [Alta] Padronizar qualidade compartilhada.
Criterios de aceite:
- ESLint, tsconfig base e convencoes compartilhadas.
- pipeline pronta para workspace.

### EPIC B2 - Extracao dos primeiros pacotes

- [ ] `INFRA-013` [Alta] Criar `packages/shared`.
Criterios de aceite:
- utilitarios puros, constants e helpers sem dependencia de app.

- [ ] `INFRA-014` [Alta] Criar `packages/ui`.
Criterios de aceite:
- primitives e componentes reutilizaveis extraidos de forma segura.
- sem dependencia de prisma, auth ou regra de negocio.

- [ ] `INFRA-015` [Alta] Criar `packages/contracts`.
Criterios de aceite:
- DTOs e schemas de fronteira consolidados.
- web e futuros consumers usando a mesma fonte.

## Fase C - Monorepo de Dominio

### EPIC C1 - Core e persistencia

- [ ] `INFRA-016` [Alta] Extrair `packages/core`.
Criterios de aceite:
- casos de uso, dominio e politicas sem dependencia de Next.js.
- features estaveis movidas sem quebrar importacao.

- [ ] `INFRA-017` [Alta] Extrair `packages/database`.
Criterios de aceite:
- Prisma schema, migrations, client e adapters centralizados.
- acesso ao banco fora do app web controlado.

- [ ] `INFRA-018` [Alta] Definir estrategia de repositories/adapters para persistencia.
Criterios de aceite:
- `core` nao depende de Prisma diretamente.
- contratos de persistencia definidos para features criticas.

### EPIC C2 - BFF e transporte

- [ ] `INFRA-019` [Media] Criar `packages/api` com roteadores modulares.
Criterios de aceite:
- tRPC/BFF consumindo casos de uso.
- authz, logging e error mapping padronizados.

- [ ] `INFRA-020` [Media] Definir padrao de data fetching e invalidacao.
Criterios de aceite:
- web deixa de depender de fluxos duplicados entre page/action/component.
