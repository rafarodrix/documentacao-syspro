## Objetivo

Concentrar apenas o backlog ainda nao concluido, priorizando:

- escalabilidade da arquitetura
- reducao de acoplamento
- qualidade de entrega
- preparacao para mobile e backend separado

## Diretriz de execucao

O backlog passa a seguir esta ordem:

1. consolidar o **modular monolith**
2. estabilizar contratos e boundaries
3. extrair o **monorepo estrutural**
4. extrair o **monorepo de dominio**
5. abrir caminho para mobile e API dedicada

Nao vamos tratar workspaces, mobile e API separada como primeiros passos.

## Fase A - Modular Monolith Disciplinado

### EPIC A2 - Actions finas e contratos internos

- [x] `INFRA-006` [Alta] Criar view models e contratos por feature dentro do app atual.
Criterios de aceite:
- dashboard, listagem e detalhes usam o mesmo contrato.
- nenhum componente forte depende de shape improvisado local.
Status de fechamento:
- `tickets` passou a centralizar contratos de listagem, detalhe, paginação, status e dashboard em `src/features/tickets/domain/model.ts`.
- `src/components/platform/tickets/types.ts` virou apenas reexport do dominio da feature, eliminando duplicacao de contratos.
- dashboard e listagem de chamados passaram a consumir o mesmo contrato do modulo.
- painéis fiscais de anexos, crédito presumido e NCM passaram a consumir diretamente os tipos do dominio fiscal, sem tipos locais duplicados.

- [x] `INFRA-007` [Alta] Padronizar gateways/adapters tecnicos em `core/infrastructure`.
Criterios de aceite:
- Zammad, company registry, tax sync e observabilidade seguem contrato explicito.
- pagina nao fala com client tecnico direto.
Status de fechamento:
- `company-registry` ja operava com contrato explicito em `src/core/domain/interfaces/company-registry-gateway.interface.ts`.
- `Zammad` passou a expor contrato em `src/core/domain/interfaces/zammad-gateway.interface.ts`, com adapter tipado em `src/core/infrastructure/gateways/zammad-gateway.ts`.
- observabilidade Zammad passou a expor gateway tipado em `src/core/domain/interfaces/observability-gateway.interface.ts` e `src/core/infrastructure/observability/zammad-observability.ts`.
- `tax sync` passou a usar contrato explicito de payload em `src/features/tax/domain/model.ts`.
- a dashboard page deixou de importar observabilidade tecnica direto de infraestrutura e passou a consumir a feature de tickets.

## Fase C - Monorepo de Dominio

### EPIC C1 - Core e persistencia

- [ ] `INFRA-016` [Alta] Extrair `packages/core`.
Criterios de aceite:
- casos de uso, dominio e politicas sem dependencia de Next.js.
- features estaveis movidas sem quebrar importacao.

- [ ] `INFRA-017` [Alta] Extrair `packages/database`.
Criterios de aceite:
- Prisma schema, migrations, client e adapters centralizados.
- acesso ao banco fora do app web controlado.

- [ ] `INFRA-018` [Alta] Definir estrategia de repositories/adapters para persistencia.
Criterios de aceite:
- `core` nao depende de Prisma diretamente.
- contratos de persistencia definidos para features criticas.
Escopo recomendado:
- formalizar quando usar `query` direta vs `repository contract`
- definir pasta padrao para contratos de persistencia por feature
- definir pasta padrao para implementacoes Prisma/adapters tecnicos
- escolher os primeiros modulos obrigatorios: `tickets`, `company` e `tax`
Entregaveis minimos:
- regra documentada na arquitetura principal
- convencao de nomes e pastas
- lista dos fluxos que ainda podem usar Prisma direto sem violar a arquitetura
